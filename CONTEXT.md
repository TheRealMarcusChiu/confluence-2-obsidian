# confluence-2-obsidian Context

## Glossary

### Source
Confluence Cloud or Server instance, accessed via REST API.

### Sink
Obsidian vault ‚Äî a local directory of Markdown files.

## Decisions

- **Data source**: Confluence Server REST API (self-hosted, v7.13.0, https://confluence.lan/)
- **API base**: `https://confluence.lan/rest/api/`
- **Auth**: Basic Auth (username + password in config/env)
- **Scope**: specific spaces, identified by space key, passed as CLI arguments
- **Content types**: pages and attachments only (no blog posts, no comments)
- **Vault structure**: mirrors Confluence page tree at full depth. Every page is emitted as both a `.md` file AND a same-named directory at the same level (e.g., `PROJ/Architecture.md` alongside `PROJ/Architecture/`). The directory holds the page's child pages (recursively) and any per-page attachments. Leaf pages with no children and no attachments still get an empty directory ‚Äî designed for the Obsidian "Folder Notes" plugin, which treats the sibling `.md` as the folder index.
- **Language**: Python
- **HTML-to-Markdown**: custom recursive parser using BeautifulSoup; walks the Confluence XML tree in one pass, handling standard HTML elements and `ac:` macros together
- **Block spacing**: tight packing ‚Äî no blank lines between block-level elements (headings, code blocks, lists, callouts, dataview blocks, tables, etc.). Three exceptions: (1) consecutive paragraphs are separated by a blank line (Markdown requires it), (2) `^excerpt` block anchor is always followed by a blank line, and (3) every top-level `<ul>` / `<ol>` is followed by a blank line ‚Äî without it, list-following Confluence macros (`latex-block` especially, which emits `$X$\n` with no leading newline) glue onto the last list line; the blank line is uniform for paragraphs, headings, macros and other plain content. **Suppressed when the following line starts with `>`** (callout or blockquote) ‚Äî a final post-process pass detects the pattern `<list-line>\n\n><‚Ä¶>` and collapses the blank line so the callout sits flush against the last list line. (Realistic case: a `children` macro after a list emits `> [!list-indent-undo]‚Ä¶` immediately after the list; the visual indent-undo callout is meant to align flush with the list, not separated by a blank line.) **Also suppressed when the following line starts with ` ``` `** (fenced code block, typically from a document-level `code` macro between two lists) ‚Äî a parallel post-process pass detects `<list-line>\n\n` ``` ` ` and collapses the blank line so the code block sits flush against the last list line, same shape and rationale as the `>` carve-out. Nested lists (rendered via the inner recursive list helper, not via the top-level `<ul>` / `<ol>` entry point) do **not** get this trailing blank ‚Äî they continue to sit flush against their parent list item content. A `<p>` whose direct children are **Tag-only** (no direct text-node content, treating whitespace-only text and `<br>` as decorative) is **unwrapped**: the paragraph spacing (`\n\n` leading and trailing) is dropped and the children render as if `<p>` weren't there. Covers Confluence-editor "cursor-park wrap" patterns where the author typed a single inline element (`<p><code>X</code></p>`, `<p><span style="color:red;">X</span></p>`, `<p><strong>X</strong></p>`) on its own line and the editor wrapped it in `<p>`. The unwrap is structural ‚Äî surrounding `<p>`s (the ones with real text content) still emit their own `\n\n`, so an unwrapped `<p>` between two text paragraphs glues to the preceding paragraph but the following paragraph still starts on a new line (asymmetric on purpose ‚Äî the unwrap rule only suppresses the wrapping `<p>`'s OWN spacing, not the siblings'). Subsumes the older "macro-only `<p>` is unwrapped" carve-out (an `<ac:structured-macro>` is a Tag, so the broader rule fires too). **Exception**: a Tag-only `<p>` is **not** fully unwrapped when any of (a) one of its direct Tag children is a colored `<span>` (i.e. `<span>` whose `style` contains `color:`), (b) one of its direct Tag children has a nested **non-`ac:*` Tag descendant** (e.g.`<p><span color><strong>X</strong></span></p>`), or (c) the `<p>` has **multiple direct plain `<span>` children** (all uncolored, no other meaningful Tag types interspersed). The multi-plain-span case (c) catches Quora-style imported content where each chunk of inline text/image is wrapped in its own bare `<span>` (e.g. `<p><span>BEFORE</span><span><ac:image‚Ä¶/></span><span>AFTER</span></p>` ‚Äî typical of Confluence pages migrated from Quora-style sources, often paired with `q-text qu-display--*` class attributes). In all three cases the `<p>` is rendered differently from a cursor-park unwrap, but the spacing depends on what precedes it: when the previous Tag sibling (skipping whitespace text nodes) is a **heading** (`<h1>`‚Äì`<h6>`), it emits `\n` + children rendering (single newline separator, tight to the heading ‚Äî `# HEADER\nTEXT` rather than `# HEADER\n\nTEXT`); otherwise it falls through to standard `<p>` rendering (`\n\n` + stripped children, normal paragraph spacing). The heading-tight branch is load-bearing for the "Confluence editor leaves a colored-span paragraph immediately under a heading" pattern; the standard-paragraph branch is what kicks in everywhere else (after a callout, list, table, plain paragraph, document start, etc.) so the colored-span paragraph doesn't glue onto the previous block's last line. Realistic cases the standard branch fixes: (1) `<ac:structured-macro ac:name="expand"><ac:rich-text-body><p>INSIDE</p></ac:rich-text-body></ac:structured-macro><p><span color>OUTSIDE</span></p>` renders as `> [!expand]- Click here to expand...\n> INSIDE\n\nOUTSIDE` (blank line between callout and paragraph); under the old "always `\n`" rule it would have glued (`> INSIDE\nOUTSIDE`). Realistic case the heading-tight branch handles for multi-plain-span (c): (2) `<h1><span>Header</span></h1><p><span>BEFORE</span><span><ac:image‚Ä¶/></span><span>AFTER</span></p>` renders as `# Header\nBEFORE![[Page/file]]AFTER` (tight, single `\n` between heading and the Quora-style chunked paragraph) ‚Äî the multi-span carve-out emits children-rendering after the `\n`, where the bare `<span>`s unwrap transparently and the embedded image renders as a wiki-link embed inline. Targets Confluence's "colored-span wrapped paragraph" pattern ‚Äî Confluence's editor leaves `<span style="color: rgb(51,51,51);">‚Ä¶</span>` around prose when the author applied and then unapplied a color (rgb(51,51,51) is near-default text color), and that markup signals real prose rather than a cursor-park artifact. The nested-tag branch catches cases where the inner formatting is `<strong>` / `<em>` etc. even without a colored span. `ac:*` descendants (macro parameters, rich-text bodies, etc.) don't trigger the carve-out, preserving the existing macro-only-`<p>` unwrap behavior (`<ac:parameter>` / `<ac:rich-text-body>` are structural, not formatting). Realistic cases: `<h1><span color>HEADER</span></h1><p><span color>TEXT</span></p>` renders as `# <font>HEADER</font>\n<font>TEXT</font>` (heading and paragraph on adjacent lines, single `\n` between), AND `<p><span color><strong>X</strong></span></p>` between two text paragraphs produces a single-`\n`-separated line instead of merging into a flush-glued run. **Single link-or-embed carve-out**: a Tag-only `<p>` whose only direct Tag child is one of `<a>` (link), a single-line embed macro (`widget`, `view-file`, `viewpdf`, `multimedia`, `excerpt-include`), or a plain (uncolored) `<span>` is **NOT** unwrapped at all ‚Äî it renders as a normal paragraph with full `\n\n` leading and trailing spacing, same as a `<p>` containing prose. Rationale: each of these is either a real "this paragraph IS the link/embed" intent, or a transparent Confluence-editor wrapper that should pass through to standard paragraph rendering. The plain `<span>` case specifically catches `<p><span>X</span></p>` (and `<p><span>X</span></p>` with `style` attributes that don't include `color:`) ‚Äî Confluence's editor frequently wraps typed prose in a bare `<span>` after a paste or selection operation, leaving the wrapper as invisible cruft; treating it as cursor-park (the old behavior) would glue the text onto preceding content. A colored `<span>` (`style` contains `color:`) does NOT take this carve-out ‚Äî it falls into the colored-span/nested-formatting carve-out above (heading-tight or standard paragraph based on previous sibling). Without the carve-out, cursor-park unwrap glues the output onto preceding content (e.g., `<h1>HEAD</h1><p><widget url="‚Ä¶"/></p>` renders as `# HEAD![](url)` ‚Äî same line ‚Äî instead of `# HEAD\n\n![](url)`). **Block-shaped macros** (`info` / `warning` / `expand` / `code` / `excerpt` / `children` / `recently-updated` / `ui-tabs`) are explicitly EXCLUDED ‚Äî their rendered output already begins with `\n` so they self-separate from preceding content even under cursor-park unwrap; promoting them would emit a redundant blank line. `latex-inline` / `latex-block` / `latex` are also excluded ‚Äî `$X$` shape is ambiguous (standalone equation paragraph vs. cursor-parked inline math), err on cursor-park to match the existing rule for bare `<code>` / `<strong>`. Decorative `<br>`/whitespace doesn't disqualify (so `<p><widget/><br/></p>` and `<p> <a>X</a> </p>` both take the carve-out). Multi-child shapes (`<p><a>X</a> <a>Y</a></p>`, `<p><widget/><widget/></p>`, `<p><strong>X</strong><a>Y</a></p>`) stay in the standard cursor-park unwrap path ‚Äî the "single direct meaningful Tag child" predicate is narrow on purpose, since multi-element shapes are rare and ambiguous in Confluence content. Realistic cases: `<p>TODO</p><p><a href="‚Ä¶">Click here</a></p>` renders as `TODO\n\n[Click here](url)`; `<h1>VIDEO</h1><p><widget url="https://youtube.com/‚Ä¶"/></p>` renders as `# VIDEO\n\n![](https://youtube.com/‚Ä¶)`. Both were previously glued (`TODO[Click here](url)`, `# VIDEO![](https://youtube.com/‚Ä¶)`) under the old "any single inline Tag is cursor-park" rule.
- **Paragraph margin-left indent**: a `<p>` whose `style` attribute contains `margin-left: Xpx` (e.g. `style="margin-left: 40.0px;"`) is rendered as a `> [!indent]` Obsidian callout wrapping the paragraph body. Confluence emits this style when the author uses the editor's toolbar Indent button (each toolbar level steps margin-left by 40px). Per-paragraph **level** is `round(X / 40)` ‚Äî non-multiples are snapped to the nearest 40 (`30px ‚Üí 1`, `70px ‚Üí 2`, `100px ‚Üí 3`, etc.); `level < 1` (e.g. `0px`, `10px`) is treated as "no indent" and the paragraph renders as an ordinary `<p>`. Per-paragraph **depth** is computed at the document-output boundary: within a consecutive run of margin paragraphs (uninterrupted by non-margin content), the smallest level in the run is depth 1; each next-deeper level adds one `> ` prefix to the callout. Formula: `depth = current_level - min_level_in_run + 1`. Examples: run `[40, 80, 120]` produces depths `[1, 2, 3]` (nested three-level callout); run `[80]` alone produces depth `1`; run `[40, 80, 40]` produces `[1, 2, 1]`. Within a single run the nested callouts sit flush against each other (no blank lines between adjacent levels ‚Äî they form one Russian-doll structure); between separate runs, or between a callout and surrounding non-margin content, the normal paragraph-break blank line applies. Implementation is two-pass: `<p>` rendering emits a `<<INDENT:LEVEL>>body<<INDENT_END>>` sentinel; a post-process pass (`_emit_indent_callouts`) finds runs of consecutive sentinels (separated only by whitespace), computes relative depths, and rewrites each as `> ` √ó depth + `[!indent]` header line plus `> ` √ó depth-prefixed body lines. Realistic case: a Confluence page whose author repeatedly tabbed-indent to create a nested indent visual (`<p style="margin-left: 40px">ONE</p><p style="margin-left: 80px">TWO</p>‚Ä¶`) renders as a single nested callout that Obsidian visualises as a stepped indent. (Was previously rendered as `int(X / 20)` ASCII spaces, which lost the visible indent at document start due to the final `.strip()` and didn't compose well with surrounding block content ‚Äî replaced with the `[!indent]` callout for native Obsidian integration.)
- **Line breaks**: `<br/>` becomes a plain `\n` ‚Äî no trailing backslash. Visual line break is preserved in Obsidian's reading view under the default **"Strict line breaks" = OFF** setting, which converts a soft break (single source newline inside a paragraph) into a rendered `<br>`. Under strict-line-breaks ON, soft breaks render as spaces and `<br/>`-derived line breaks would be lost ‚Äî not supported, same assumption as the multi-`<code>` `<pre>` rule. Was previously a backslash hard break (`\` at end of line) which forced the `<br>` under both settings; dropped because the visible trailing `\` made source `.md` files noisy and some editors / linters strip trailing whitespace adjacent to it. Inside a `<pre>` block, `<br/>` is also `\n` (same handler) and feeds the multi-`<code>` line splitter directly. **Inside a depth-0 `<li>`'s inline body** (the rendered content that becomes the list-item's marker line ‚Äî first non-empty `<p>`, plus any raw text or inline tags around it that aren't hoisted as block content), every `\n` in the body ‚Äî typically from `<br/>`, but the rule fires on any newline in the inline content ‚Äî picks up a continuation indent matching the marker's text-start column so the multi-line body stays inside the same list item under CommonMark. The indent is `' ' * (len(marker) + 1)`: depth-0 `-` items continue with 2 spaces, depth-0 `1.` with 3 spaces (`10.` with 4 spaces, scaling with the marker width). **At depth ‚â• 1** (nested `<li>`s) the continuation is dropped ‚Äî `<br/>` produces a bare `\n` and the post-`<br/>` content sits at column 0, effectively terminating the inner list at the `<br/>` boundary. The rationale: when a nested list item's body splits over multiple lines, the natural visual intent is for the continuation to break out of the list (rather than carry the inner indent + continuation offset, which Obsidian doesn't render cleanly for the deeply-indented case). (Was previously a hard collapse of `\n` to a single space at all depths; depth-0 continuation indent was added, then narrowed to depth-0-only after observing the column-0 break is what nested lists actually want.)
- **List nesting indent**: nested lists are indented with one tab character per level (not spaces).
- **Adjacent ordered-list break**: when two ordered lists end up back-to-back in the rendered markdown ‚Äî same indent depth, no separating block between them ‚Äî a single-line empty blockquote (`> ` at the matching indent, surrounded by blank lines) is inserted between them so they render as two distinct lists rather than one merged sequence. Trigger: at any indent depth `N`, an ordered list line (`\d+\. ‚Ä¶`) immediately followed (one or more newlines later) by a line that starts with `1\. ` at the same depth ‚Äî the `1` is the tell-tale "new list restarts numbering" signal, since the converter always emits each `<ol>`'s items starting at 1. Injection: replace the `\n+` gap with `\n\n<indent>> \n\n` so the separator's indent matches the surrounding lists, keeping it nested-safe. Why an empty blockquote: CommonMark merges consecutive ordered lists into one (renumbering 1, 2, 3, ‚Ä¶ across the merge) and a blank line alone doesn't break them; a different block element between them does, and `> ` is the lightest visible option (single character + space). Realistic source: in this corpus the trigger only fires when Confluence emits a cursor-park `<p><br/></p>` between two `<ol>` siblings ‚Äî the cursor-park is dropped to empty (per the `<p>` rule above), the two lists land adjacent in the output, and the post-process injects the breaker. Detection lives at the output boundary (post-process) rather than at the `<p>` site so the rule catches the bad adjacency regardless of what source pattern produced it.
- **Cursor-park `<p><br/></p>` as same-type list separator**: a `<p>` whose only non-whitespace content is `<br/>` (Confluence's cursor-park artifact, normally dropped to empty per the empty-inner rule) is **kept** and emits a literal `<br>` on its own line when it sits between two same-type list siblings ‚Äî both `<ul>` or both `<ol>` ‚Äî at the document level. Detection lives at the `<p>` rendering site: walk `previous_sibling` and `next_sibling` (skipping whitespace text nodes); fire only when both immediate non-whitespace siblings are Tags with the same name AND that name is `ul` or `ol`. The `<br>` line is a CommonMark "type 6" HTML block (terminated by the next blank line); on its own line at column 0 it breaks the otherwise-automatic merge of consecutive same-marker lists (`- ` / `- ` would merge into one bulleted list; `1. ` / `1. ` would renumber across the merge). Mixed-type adjacency (`<ul>` ‚Üî `<ol>`) and list-followed-by-other-block (`<ul>` ‚Üî `<h1>`, etc.) don't trigger ‚Äî the cursor-park drops to empty as before, since no merge problem exists. Realistic case: `<ul><li>BEFORE</li></ul><p><br/></p><ul><li>AFTER</li></ul>` renders as `- BEFORE\n<br>\n- AFTER` (two distinct lists); without the carve-out the cursor-park would drop and `- BEFORE\n\n- AFTER` would render as one merged 2-item list. Composes with the existing **Adjacent ordered-list break** rule (`> ` blockquote): when the source has `<p><br/></p>` between two `<ol>`s, this carve-out fires first (`<br>` emitted at the `<p>` site); the output then no longer matches the `\d+\. ‚Ä¶\n+1\. ` adjacency regex, so the `> ` blockquote is NOT injected ‚Äî both lists already render distinct. The `> ` blockquote rule still handles the rarer case where two `<ol>`s end up output-adjacent without a `<p><br/></p>` source between them.
- **List depth-jump normalization**: after the converter produces the markdown output, a final post-process pass scans for list lines whose indent depth exceeds the natural progression and injects empty placeholder list items at every missing depth so the indentation chain is unbroken from depth 0 to the deepest line. A "list line" is any line matching `^\t*- ` or `^\t*\d+\. `; its **depth** is the number of leading tabs. The pass maintains a running stack of "live" depths as it walks line-by-line: a list line at depth N must have parents at every depth `0..N-1` already in the stack ‚Äî if any are missing, a bare `- ` placeholder is injected at each missing depth before the line is emitted, then the stack is trimmed to `0..N`. Any non-list line (plain text, blank line, heading, callout, code fence, table HTML, etc.) **resets the stack** ‚Äî the next list line starts a fresh hierarchy. Placeholder marker is **always `-`**, never `1.`, regardless of the deeper line's style: ordered lists with depth jumps get bulleted placeholders (`1. foo` at depth 2 with no parents ‚Üí `- \n\t- \n\t\t1. foo`) ‚Äî the placeholder has no content so the "1" numbering would be meaningless, and a single placeholder style is easier to scan visually. The deeper line's marker is preserved verbatim (no renumbering, no ordered‚Üíbulleted conversion). The transformation is **idempotent**: applying it to already-valid markdown produces no change. Defensive: catches depth-jump cases regardless of which Confluence source pattern triggered them (rare in practice, but the renderer's recursion plus block-hoisting interaction makes the invariant worth enforcing at the output boundary rather than trusting every path through `_render_list_inner` to preserve it).
- **`<p>` inside `<li>`**: the first `<p>` child of an `<li>` **whose rendered content is non-empty** is treated as a paragraph-break **split point**: its inline content becomes the list item's inline body, and every sibling that appears *after* that `<p>` (whether `<p>`, raw text, inline tag, or block-emitting tag) is **hoisted** out and rendered at column 0 on the line immediately following the list item's inline content (same hoist mechanism as the existing "block content inside `<li>`" rule). Without this rule the post-`<p>` siblings glue onto the list item line with no separator: `<li><p>hello world</p>after-text</li>` renders as `- hello worldafter-text` and `<li><p>X</p><strong>Y</strong></li>` renders as `- X**Y**`. The cause is that `<p>` renders to `\n\nX` whose stripped form has no embedded newline, so the existing block-content hoist (which keys on stripped output containing `\n`) doesn't fire ‚Äî the `<p>` body and surrounding siblings all land in `inline_parts`, and the body builder strips leading whitespace, collapsing the separation. The `<p>`-split rule fires for any `<p>` child regardless of class ‚Äî `auto-cursor-target` (Confluence's auto-generated-paragraph marker) is the most common but not the only trigger. **Empty `<p>`s** (cursor-park `<p></p>` / `<p><br/></p>` Confluence inserts around real content) are skipped ‚Äî they don't claim the split, so the next sibling is free to land as inline content on the list-item line. Without this carve-out, `<li><p><br/></p><latex-inline/></li>` would render as a bare `- ` line followed by a hoisted `$X$` on the next line; with it, the latex stays inline on the marker line: `- $X$`. Content *before* the first non-empty `<p>` (rare in practice ‚Äî Confluence's editor wraps typed text in `<p>` automatically) joins the inline body via existing inline_parts concatenation.
- **Block content inside `<li>`**: when a list item contains a child whose rendered output spans multiple lines (`expand` / `ui-expand` / `info` / `warning` callouts, multi-line `<table>` HTML, etc., detected as any rendered child whose stripped output contains a `\n`), the block is hoisted out of the list item and emitted at column 0 on the line immediately following the list item's inline content. **Exception: fenced code blocks** (the Confluence `code` macro and `merge-table` tables ‚Äî both produce rendered output starting with ` ``` `, detected by `content.startswith('‚Äã```')` at the hoist site) are **not** hoisted to column 0 ‚Äî instead each line of the fenced block is prefixed with `'\t' * depth + '    '` (depth-0 list at top: 4 spaces; depth-N nested: tab√óN + 4 spaces) so the block stays inside the list item as proper CommonMark list-item content. No warning is logged for hoisting at all ‚Äî the original Confluence relationship ("this block belongs to this list item") is lost in conversion, but spamming the migration report with one warning per occurrence produced too much noise to be useful; the cleanup signal now lives in the visible output (an orphan `- ` line above a hoisted block reads as obviously wrong). Scope: document level only ‚Äî lists inside table cells go through the cell-content render path which preserves them as raw HTML, so the rule doesn't fire there. Separator before the indented fence is always a single `\n` (no blank line) regardless of preceding content. **Hoisted blocks and nested lists are interleaved in source order** ‚Äî the `<li>`'s children are walked once, and each non-inline child (multi-line block, nested `<ul>` / `<ol>`, or post-first-`<p>` sibling) is emitted in the position it appeared in source, not segregated into "blocks first, nested last" as before. Adjacent hoisted blocks (both at column 0) are separated by a blank line so Obsidian renders them as separate blocks ‚Äî **except** when the previous block ends with `</span>` (typically the `<pre>`-rendered `<span style="white-space: pre-wrap">` wrapper, which is inline HTML and doesn't need a blank-line terminator); in that case the separator drops to a single newline so the wrapper sits flush against the following hoisted content. A nested `<ul>` / `<ol>` joins to the preceding sibling with a single newline (no blank line) regardless. **Hoisted blocks that appear AFTER a nested list in source order get a 2-space indent prefix** on every line, with a blank line separator above ‚Äî visually aligning with the depth-0 `- ` placeholder column that the depth-jump normalization pass injects before the nested list. Without this offset the trailing content would sit at column 0 and look detached from the list item it belongs to. Scope is the in-`<li>` hoist boundary only ‚Äî the rule does NOT apply at the document-level boundary between the closing `</ul>` / `</ol>` and the next sibling block, where the standard list-trailing blank line (rule (4) of the Block-spacing decision above) still fires and the next paragraph lands at column 0. Realistic cases: (a) `<li><p>BEFORE</p><pre>HELLO</pre><p>AFTER</p></li>` ‚Äî the `<pre>` wrapper and the trailing `<p>` are both hoisted and joined with `\n` (per the `</span>` carve-out above), output `- BEFORE\n<span>...</span>\nAFTER`. (b) `<li><p>WORLD</p><table>‚Ä¶</table><ul><li>HELLO</li></ul><p>AFTER</p></li>` (Confluence's typical "ordered list with embedded table, nested bullets, then trailing prose" shape) ‚Äî output is `1. WORLD\n    ```merge-table\n    ‚Ä¶\n    ```\n- \n\t- HELLO\n\n  AFTER`: marker line, then indented merge-table fence (inside the list item via the fence exception above, no hoist), then nested list (with depth-jump `- ` placeholder injected at depth 0, because the intervening fence lines are non-list lines that reset the depth-jump pass's running stack), then trailing `<p>` with the 2-space post-nested indent. The list is implicitly terminated at column 0 ‚Äî the hoisted block renders as a top-level element, not as visual content of the list item. The list marker line **always keeps its trailing space** after the marker (`- ` for bullets, `1. ` for ordered) even when the list item's inline content is empty ‚Äî CommonMark requires a space after the marker for it to be recognized as a list bullet (a bare `-` on its own line parses as plain text, not a list item), so the empty-body-followed-by-hoisted-block case (e.g. an `<li>` containing only an `expand` macro) emits `- ` on the marker line and the hoisted callout on the next line, preserving the visual "empty bullet + block" intent.
- **Inline HTML preserved**: `<sub>` and `<sup>` are emitted as raw HTML (`<sub>2</sub>`, `<sup>2</sup>`) ‚Äî Markdown has no native equivalent and Obsidian renders them in reading view. Same approach as the existing `<u>` handling. `<strong>` / `<b>` and `<em>` / `<i>` are **also** emitted as raw HTML (`<strong>foo</strong>`, `<em>foo</em>`) rather than as their Markdown equivalents (`**foo**`, `*foo*`) ‚Äî Markdown bold/italic are fragile under several common patterns (e.g., `<strong>X</strong><strong>Y</strong>` becomes `**X****Y**` which Markdown parses ambiguously; an isolated `<strong> </strong>` becomes `** **` which renders as literal asterisks with a space). Raw HTML survives all those cases cleanly because Obsidian renders the tags directly. **Whitespace-only carve-out**: when the inner rendered content of a `<strong>` / `<b>` / `<em>` / `<i>` is whitespace-only (or absent entirely), the tag is dropped and the inner whitespace emitted directly ‚Äî so `<strong> </strong>` becomes a single space, `<strong></strong>` becomes nothing, and nested `<strong><em> </em></strong>` collapses recursively to a single space. Confluence's editor frequently emits these empty-formatting tags when a user toggles bold/italic while their cursor is over a space character; the formatting carries no semantic content, and preserving the wrapper produces visual noise in the output. The carve-out does NOT apply to `<u>` / `<sub>` / `<sup>` ‚Äî those are kept verbatim even when whitespace-only (rare, and they're already much less common in Confluence content). **Boundary-`\n` strip** (separate from the whitespace-only carve-out): when the inner rendered content of a `<strong>` / `<b>` / `<em>` / `<i>` has leading or trailing `\n` characters (typically from a `<br/>` Confluence's editor leaves at the start or end of the formatting span ‚Äî e.g., `<em>ONE<br/></em>` produces inner `ONE\n`), those boundary `\n`s are stripped before the wrapper is applied. Output: `<em>ONE</em>` (the trailing `<br/>` is dropped). Mid-content `<br/>` (`<em>line1<br/>line2</em>`) is preserved verbatim ‚Äî only the leading/trailing `\n`s strip. Real whitespace (ASCII space, tab) at the boundary is NOT stripped (`<strong> world. </strong>` still renders with surrounding spaces preserved) ‚Äî only `\n` strips, so the existing syntax-preservation cases survive. Rationale: a `<br/>` at the boundary of an inline formatting tag is almost always Confluence editor cruft (the editor leaves a `<br/>` behind when the user toggles bold/italic and then presses Enter or moves the cursor out of the span), and the resulting `\n` inside the formatting tag visibly clutters output AND can trigger spurious "block-content unwrap" decisions in the colored-`<span>` rule (because a `\n` inside the inner makes the colored span think its content is block-shaped and drop the wrapper). Stripping the boundary `\n` makes the formatting wrapper survive cleanly and composes correctly with the colored-span rule. The strip is targeted at `\n` specifically (not all whitespace) ‚Äî so the existing rule "`<strong> world. </strong>` preserves boundary spaces" still holds.

The escape over-covers a few low-risk positional cases ‚Äî intraword `_` and mid-line `#` are technically safe in CommonMark, but are escaped anyway for simplicity and to defeat Obsidian's tag extension (`#123` would otherwise become a tag). Source `.md` files will show backslash-prefixed characters more often as a result; rendered output is unchanged. Applied to text nodes whose ancestor chain includes any of `<p>`, `<h1>`‚Äì`<h6>`, `<li>`, `<strong>`, `<em>`, `<u>`, `<sub>`, `<sup>`, or any `<span>` (colored or uncolored ‚Äî colored spans become `<font>` HTML in the output and are escaped on the same grounds as `<code>` / `<pre>` below). Inside `<code>` and `<pre>` (HTML inline-code and the multi-`<code>`-rendered pre block), the **same full plain-text escape set applies** (no narrowing) ‚Äî Obsidian's HTML extension processes Markdown inside HTML inline-code, so `*foo*` would otherwise render as italics and `**foo**` as bold even between `<code>` tags; `<>` are also parsed as HTML-tag boundaries by Obsidian's HTML parser; and `#word` is picked up by Obsidian's tag extension. All 11 plain-text metacharacters (`\`, `` ` ``, `*`, `_`, `~`, `#`, `[`, `]`, `$`, `<`, `>`) are backslash-escaped to render as literal characters. (Was previously narrowed to `<>#` based on a now-corrected belief that Obsidian didn't process Markdown inside `<code>`; observed `**bold**` / `*italic*` rendering inside `<code>` triggered the unification with the plain-text set.) **The same full escape applies inside `<font>`** (the rendered output for colored `<span>`) ‚Äî Obsidian's HTML extension processes Markdown inside `<font>` just as inside `<code>`, so colored spans now get the same full plain-text escape rather than the previously-applied no-escape treatment. (Was previously \"verbatim inside colored spans\" based on the same now-corrected belief.) Inside `<a>` and `<ac:link>` (link display text), only `[`, `]`, `<`, and `>` are escaped ‚Äî the other characters stay verbatim so inline formatting and math survive inside the link (e.g. `[**bold link**](url)` and `[$x^2$](url)` still render correctly), while the brackets escape so a user-typed `[draft]` inside link text doesn't break the constructed `[text](url)` parse into `[[draft]](url)` and the angle brackets escape so a user-typed `<tag>` in link text doesn't get parsed as inline HTML. Table-cell text follows the separate Tables rule below (no plain-text escape applied). Already-escaped sequences are not detected: a Confluence-typed literal `\]` becomes `\\\]` (four characters) in the output, which Obsidian renders back to `\]` ‚Äî preserving the original two-character text.

- **Block `<pre>`**: not a code block ‚Äî emitted as a `<span style="white-space: pre-wrap">` wrapper containing a sequence of inline `<code>` elements, one per rendered line, **joined by literal `<br>` HTML tags** (e.g. `<span style="white-space: pre-wrap"><code>line1</code><br><code>line2</code><br><code>line3</code></span>` on a single source line). Non-empty `<ac:link>` variants ‚Äî including unsupported target types like `<ri:user>` and `<ri:space>`, which currently render to empty text ‚Äî are unaffected and keep their existing renders. Detection is structural (`not list(tag.children)`), not content-based: whitespace-only bodies (`<ac:link>   </ac:link>`) are NOT treated as empty.
- **External URL link** ‚Üí `[Display Text](url)` standard Markdown link
- **YouTube embed** (`<ri:url>`) ‚Üí `![](url)` Media Extended plugin syntax
- **URL paren-balance encoding** (applies to every `[‚Ä¶](url)` and `![‚Ä¶](url)` emission ‚Äî `<a href>`, `<ac:link>` with `<ri:url>` target, and the `widget` macro): unbalanced parentheses in the URL portion are percent-encoded (`(` ‚Üí `%28`, `)` ‚Üí `%29`) so the Markdown parser doesn't terminate the link prematurely. Algorithm: walk the URL left-to-right tracking an "unmatched `(`" counter ‚Äî each `(` increments it; each `)` either decrements (matched, kept verbatim) or, when the counter is zero, gets encoded to `%29` as excess. Then a right-to-left pass encodes any leftover unmatched `(` to `%28`. Matched pairs survive intact so the URL stays human-readable. Realistic case: a Wikipedia URL with a `:~:text=‚Ä¶` text fragment that contains `one).%5B1%5D-` (a bare `)` after "one" that doesn't pair with anything earlier in the URL) ‚Äî the page title fragment `_(topological_vector_space)` IS a balanced pair and stays as `(topological_vector_space)`, but the unmatched `)` after `one` gets encoded to `%29` so the Markdown link terminates at its true end rather than at the fragment's `)`. The display text portion (`[‚Ä¶]` part) is NOT paren-encoded ‚Äî Markdown's link syntax doesn't treat parens as special inside `[ ]`, so author-typed parens in link text stay verbatim (existing `BRACKETS_ONLY_ESCAPE_RE` handles `[` / `]` / `<` / `>` only).
- **Attachments**: downloaded into per-page subfolder `<Space>/<Parent>/<PageName>/<filename>`. **Embeds** ‚Äî inline images (`<ac:image><ri:attachment>`), `view-file` / `viewpdf` / `multimedia` macros ‚Äî are emitted as `![[PageName/filename]]` (per-page-prefixed wiki-link embed; the prefix matches the on-disk path so the embed resolves directly to the file owned by the current page). **Links** ‚Äî `<ac:link><ri:attachment ri:filename="X"/>` ‚Äî are emitted as `[[X]]` (or `[[X|<display>]]` when an `<ac:plain-text-link-body>` CDATA is present), **without** the per-page prefix; Obsidian resolves wiki-links by name across the whole vault, so a bare-filename link finds the file regardless of which page-subfolder it lives in. **Collision trade-off** when the same filename appears in multiple per-page subfolders (e.g., two pages each with their own `diagram.png`): all `<ac:link>` references to that filename render as `[[diagram.png]]` and Obsidian resolves to whichever copy it picks first (typically alphabetical folder order). Accepted because the bare-filename form matches the user-typed convention for "link to a known file" in Obsidian; the per-page prefix is reserved for embeds, where the visual content is conceptually tied to the page that owns it. No collision detection or migration-report logging ‚Äî the destination vault owns resolution. Inline-image sizes carry over via the **Obsidian wiki-link size suffix** only when the source `<ac:image>` carries an `ac:width` attribute: `ac:width="W"` alone ‚Üí `![[PageName/filename|W]]` (Obsidian scales height proportionally); `ac:width="W"` plus `ac:height="H"` ‚Üí `![[PageName/filename|WxH]]`; neither attribute (or `ac:height` alone) ‚Üí bare `![[PageName/filename]]`. `ac:height` is intentionally **ignored when `ac:width` is absent** ‚Äî Obsidian's `|0xH` "height-only" convention does not render reliably in practice (image either renders unsized or with the wrong aspect ratio), so the safer fallback is to drop the size hint and let Obsidian display the image at its natural size. Values are read verbatim from the attributes ‚Äî no parsing, casting, or unit-stripping (Confluence's documented format is integer pixels; any non-integer value passes through and may render wrong). The `ac:style` attribute (e.g. `max-height: 250.0px;`) is **never** parsed ‚Äî Confluence's editor sets `ac:width` / `ac:height` alongside any inline style, so attribute-only reading is sufficient in practice. External-URL images (`<ac:image><ri:url>`) stay as `![](url)` with no size ‚Äî Markdown image syntax has no native size suffix. **Image inside `<li>`** (any depth) gets the same callout-wrap treatment as a Children Display macro inside `<li>`: the `![[PageName/file|W]]` wiki-link is wrapped in a doubly-nested `> [!list-indent-undo]\n> > [!indent]\n> > ![[PageName/file|W]]` callout and hoisted to column 0 below the list item, so it visually aligns flush with the list bullet (requires the same custom CSS on `[data-callout="list-indent-undo"]` and `[data-callout="indent"]` as the children-macro rule). Images at document level (no `<li>` ancestor) emit unchanged ‚Äî bare `![[PageName/file|W]]` with no callout wrap. External-URL images (`<ac:image><ri:url>`) are unaffected by the wrap. Any Unicode whitespace in the filename is normalized **one-to-one** to ASCII space `U+0020` (same `\s`-to-space pattern used for `ac:image` in table cells and for latex bodies) at **both** sites that touch the filename: every wiki-link reference emitted by the converter (embed or link), **and** the on-disk filename when the attachment is written. Both must agree ‚Äî Confluence's editor sometimes embeds NBSP (`U+00A0`) etc. in attachment filenames (most common: screenshots whose default names contain NBSP-separated timestamps), and without paired normalization Obsidian can't resolve the link to the file on disk. Applies to all reference paths: inline `<ac:image>`, attachment `<ac:link>`, and the `view-file` / `viewpdf` / `multimedia` macros. (Was previously `[[PageName/filename]]` for `<ac:link>` attachment too ‚Äî dropped the prefix because the user-typed convention for "link to a known file" in Obsidian is the bare filename, and the per-page prefix on links was redundant given Obsidian's name-based resolution. Embeds keep the prefix because they anchor visual content to the owning page; a link is a navigation target, an embed is the content itself.)
- **Unknown macros**: logged and skipped; a `migration-report.md` written to vault root with macro name, page, and count
- **Tables**: every `<table>` emits a fenced ` ```merge-table ` block containing a single JSON object ‚Äî rendered by the **Merge Table Obsidian plugin** (maintained alongside this codebase). The plugin renders Markdown natively inside cells (bold, italic, code blocks, lists, callouts, wiki-links, images, nested tables ‚Äî anything the document-level Markdown renderer produces), eliminating the cleaned-raw-HTML workaround the previous design used to escape Markdown's "no rendering inside `<table>`" constraint. JSON shape: top-level `{"rows": [[...], ...], "tableStyle"?: "..."}`. `<caption>` elements are dropped silently (the format supports a `caption` key but the corpus rarely uses it). `<colgroup>` / `<col>` are stripped (no merge-table equivalent). `<thead>` / `<tbody>` / `<tfoot>` are pass-through wrappers ‚Äî their `<tr>` children feed the flat `rows` list in source order. Each row entry is a shorthand string `"‚Ä¶"` (prose-only cell, equivalent to `{"content": "‚Ä¶"}`), a full object `{"content": ..., "header"?: true, "bg"?, "color"?, "align"?, "valign"?, "colspan"?, "rowspan"?}`, or `null` (placeholder for a colspan-covered position in the same row). Empty `<td>` ‚Üí `""` shorthand. Cells with `colspan="N"` are followed by `N-1` `null` placeholders in the same row (visual grid alignment for source `.md` readability; the plugin doesn't require them). `rowspan="N"` covered positions in later rows are auto-skipped (no `null` emitted ‚Äî the plugin handles those automatically). **Attribute extraction**: `<th>` ‚Üí `header: true`, plus auto `bg: "#F4F5F7"` when no explicit background is set (preserves Confluence's default header tint; the plugin's `header: true` default style may not match Confluence's). `data-highlight-colour="X"` ‚Üí `bg: "X"` (the colour name `grey` is remapped to `#F4F5F7`); takes precedence over any `background-color:` in `style`. `style="background-color: X"` ‚Üí `bg: "X"` when `data-highlight-colour` is absent. `style="color: X"` ‚Üí `color: "X"`. `style="text-align: X"` ‚Üí `align: "X"`, filtered to `{left, center, right}` (Confluence sometimes emits `justify` / `start` / `end` ‚Äî dropped, no `align` key emitted). `style="vertical-align: X"` ‚Üí `valign: "X"`, filtered to `{top, middle, bottom}`. `colspan="N"` / `rowspan="N"` ‚Üí `colspan: N` / `rowspan: N` (cast to int). All other `style` declarations on cells (`width`, `padding`, `font-family`, `border`, etc.) are dropped ‚Äî merge-table has no per-cell CSS escape hatch, and Obsidian renders tables responsively. Table-level `<table style="...">` flows through as top-level `tableStyle: "..."` (verbatim). **Cell content (`content` key)**: the cell's children are rendered through the **full document-mode Markdown converter** ‚Äî paragraphs, lists, code blocks, callouts (`> [!info]` / `> [!warning]` / `> [!expand]-`), latex (`$X$`), wiki-links (`[[Page]]`), images (`![[Page/filename|WxH]]`), and other macros all produce their native Markdown forms. Walk the cell's children once; accumulate Markdown until a nested `<table>` is encountered; emit the accumulated string + the nested table object; reset and continue. Three output shapes by what's in the cell: prose-only (no nested table) ‚Üí `"content": "<markdown string>"`; a single nested `<table>` with no surrounding prose ‚Üí `"content": {"rows": [...]}` object form; mixed Markdown + nested table(s) ‚Üí `"content": ["<md>", {"rows": ...}, "<md>", ...]` array form (interleaved in source order, per the README's array-`content` shape). An empty cell shortcuts to bare `""` shorthand (no `"content"` wrapping). **Unknown macros inside cells** flow through document-mode's standard handling ‚Äî logged to `unknown_macros` (which surfaces in `migration-report.md`) and dropped, no visible XML in the rendered cell. Behavior change vs. the previous cleaned-HTML design, which escaped unknown `<ac:*>` to literal source XML in the cell as a manual-cleanup signal; with merge-table, the cleanup signal is the report entry, not the rendered cell. **`latex-block` in cells** still emits the `latex-block rendered as inline math on page X` warning (rendered as `$X$\n` via the document-mode latex handler ‚Äî inline math, not Obsidian block math `$$X$$`). **`<ac:image>` in cells** uses the document-mode handler (`![[PageName/filename|WxH]]` Obsidian embed) instead of the previous bare `<img src="filename" />` ‚Äî links resolve via the per-page subfolder and the size suffix carries over from `ac:width` / `ac:height` (was previously a bare filename in `<img>` with no subfolder prefix, requiring manual fixup; the document-mode handler does the right thing for free). **JSON formatting**: `json.dumps(table_def, indent=2, ensure_ascii=False)` ‚Äî standard 2-space pretty-print, Unicode kept literal (Asian / emoji content stays human-readable in source `.md`). Output is deterministic. Doesn't reproduce the README's tighter idiom (rows-of-strings inline, rows-with-objects multi-line); the plugin parses identically and the readability cost is acceptable. **Block spacing**: the fenced ` ```merge-table ` block emits with `\n\n` leading whitespace (matching the callout convention ‚Äî `info` / `warning` / `expand` all use `\n\n` lead). When preceded by a paragraph (which itself emits `\n\n` leading + text), the combined whitespace + `\n{3,}`-collapse produces exactly one blank line between the paragraph and the opening fence ‚Äî required by readability if not strictly by CommonMark (a fenced code block CAN interrupt a paragraph without a blank line, but the visual separation is cleaner). Trailing side: no explicit trailing whitespace from `_render_table`; the following block provides its own leading spacing (a standard `<p>` after the table emits `\n\n`, producing a blank line). When preceded by a heading (`<h1>`‚Äì`<h6>` emit `\n` leading only), the table's `\n\n` leading produces a blank line between heading and table ‚Äî a behavior change from the previous "tight after heading" rule but acceptable since the corpus has no heading-immediately-followed-by-table cases that need to stay tight. No special `</table>` carve-out is needed (no `</table>` appears in the output). **Inside a `<li>`**: detected as a fenced block via the existing `content.startswith('‚Äã```')` rule (same path as the Confluence `code` macro), indented with `'\t' * depth + '    '` per line so the table stays inside the list item ‚Äî no hoist to column 0 (preserves the visual relationship to the parent list item). Other multi-line block constructs hoisted from `<li>` (raw `<pre>` wrappers, etc.) still hoist to column 0 as before, but no warning is logged (see the **Block content inside `<li>`** decision above). (Was previously cleaned + prettified raw HTML emission with ~10 cell-context special cases ‚Äî `<details><summary>` for expand, `<span class="math display">` for latex, multi-`<code>`+`<br>` for code, bare `<img>` for images, plain-text fallback for `<ac:link>`, escape-as-source-XML for unknown `<ac:*>`, etc. ‚Äî all subsumed by the document-mode-end-to-end design once the Merge Table plugin made Markdown rendering inside cells possible.)
- **Filename sanitization**: characters illegal in filenames (`:`, `/`, `?`, `*`, `<`, `>`, `|`, `\`) in page titles are replaced with their visually similar Unicode fullwidth equivalents (`ÔºöÔºèÔºüÔºäÔºúÔºûÔΩúÔºº`) when used as file or directory names ‚Äî preserves the appearance and is lossless (no two distinct Confluence titles collapse to the same on-disk name); post-sanitization collisions are still resolved by suffixing the Confluence page ID and logged in `migration-report.md`
- **Migration mode**: one-time, full re-run on every invocation (no incremental sync, no resume); Ctrl+C triggers a graceful stop that writes `migration-report.md` before exiting
- **migration-report.md**: written to vault root on graceful stop, crash, or completion; contains all unknown macros encountered (macro name, page title, count) and all per-page failures (page title, error reason)
- **Frontmatter**: created, modified, labels (as Obsidian tags), **parent** (omitted on root pages with no ancestors), **children** (always emitted ‚Äî empty list `[]` on leaf pages). Both `parent` and `children` use the **sanitized on-disk basename wrapped in `[[‚Ä¶]]`** (Obsidian wiki-link form) ‚Äî e.g. `parent: "[[Architecture]]"`, `children: - "[[Module A]]"`. The wiki-link wrapping makes the property values clickable in Obsidian's reading view and surfaces them to Dataview as link objects (rather than opaque strings). The basename inside the brackets is the same form that appears in the `<title>.md` filename ‚Äî **without** the `.md` extension ‚Äî matching Obsidian's wiki-link resolution. YAML-quoted as `"[[X]]"` (unquoted `[[X]]` would be invalid YAML ‚Äî `[` starts a flow sequence). Empty `children` stays as `children: []` (nothing to wrap). The parent ‚Üí children relationship is derived once during migration from the `ancestors` array of every page in the page set; each page's `children` is the set of OTHER pages whose direct parent (`ancestors[-1]`) is this page (kept stable across runs via alphabetical sort). Pages outside the current migration scope (e.g., a parent in a space that wasn't migrated, or children excluded by the space-key filter) are NOT followed ‚Äî `parent` may be omitted even on a non-root page if its parent isn't in the migration set, and `children` only includes pages that were actually migrated.
- **Config**: `.env` file (`CONFLUENCE_URL`, `CONFLUENCE_USER`, `CONFLUENCE_PASS`, `VAULT_PATH`, `CONFLUENCE_VERIFY_SSL`, `DOWNLOAD_ATTACHMENTS`); space keys passed as CLI args
- **Progress display**: each page migration prints `[<n>/<total>] <Page Title>` to stdout
- **SSL**: verification skipped (`verify=False`), warnings suppressed; controlled by `CONFLUENCE_VERIFY_SSL` env var
- **Attachment download**: controlled by `DOWNLOAD_ATTACHMENTS` env var (default `true`); when `false`, attachments are skipped and attachment references are emitted as broken wiki-links ‚Äî `![[PageName/diagram.png]]` for embeds (inline images, `view-file` / `viewpdf` / `multimedia` macros), `[[diagram.png]]` (or `[[diagram.png|<display>]]`) for `<ac:link>` attachment links ‚Äî both unresolvable since no file is present in the vault
