# confluence-2-obsidian Context

## Glossary

### Source
Confluence Cloud or Server instance, accessed via REST API.

### Sink
Obsidian vault — a local directory of Markdown files.

## Decisions

- **Data source**: Confluence Server REST API (self-hosted, v7.13.0, https://confluence.lan/)
- **API base**: `https://confluence.lan/rest/api/`
- **Auth**: Basic Auth (username + password in config/env)
- **Scope**: specific spaces, identified by space key, passed as CLI arguments
- **Content types**: pages and attachments only (no blog posts, no comments)
- **Vault structure**: mirrors Confluence page tree at full depth. Every page is emitted as both a `.md` file AND a same-named directory at the same level (e.g., `PROJ/Architecture.md` alongside `PROJ/Architecture/`). The directory holds the page's child pages (recursively) and any per-page attachments. Leaf pages with no children and no attachments still get an empty directory — designed for the Obsidian "Folder Notes" plugin, which treats the sibling `.md` as the folder index.
- **Language**: Python
- **HTML-to-Markdown**: custom recursive parser using BeautifulSoup; walks the Confluence XML tree in one pass, handling standard HTML elements and `ac:` macros together
- **Block spacing**: tight packing — no blank lines between block-level elements (headings, code blocks, lists, callouts, dataview blocks, tables, etc.). Four exceptions: (1) consecutive paragraphs are separated by a blank line (Markdown requires it), (2) `^excerpt` block anchor is always followed by a blank line, (3) every `</table>` is followed by a blank line — required for the HTML block to terminate so subsequent Markdown parses (CommonMark §4.6); suppressed for tables nested inside another `<td>`/`<th>` (must sit flush per the Tables rule); **also suppressed at the top level when the next content is a plain-text paragraph** — a final post-process pass detects the pattern `</table>\n\n<plain-text-line>` and collapses to `</table>\n<plain-text-line>` so a table flowed by ordinary prose sits flush (the blank is preserved when the following line starts with a Markdown block marker — `#`, `-`, `\d+. `, `>`, `` ` ``, `^`, or `<` — because those still need the blank to round-trip through CommonMark's HTML-block termination), and (4) every top-level `<ul>` / `<ol>` is followed by a blank line — without it, list-following Confluence macros (`latex-block` especially, which emits `$X$\n` with no leading newline) glue onto the last list line; the blank line is uniform for paragraphs, headings, macros and other plain content. **Suppressed when the following line starts with `>`** (callout or blockquote) — a final post-process pass detects the pattern `<list-line>\n\n><…>` and collapses the blank line so the callout sits flush against the last list line. (Realistic case: a `children` macro after a list emits `> [!list-indent-undo]…` immediately after the list; the visual indent-undo callout is meant to align flush with the list, not separated by a blank line.) Nested lists (rendered via the inner recursive list helper, not via the top-level `<ul>` / `<ol>` entry point) do **not** get this trailing blank — they continue to sit flush against their parent list item content. A `<p>` whose direct children are **Tag-only** (no direct text-node content, treating whitespace-only text and `<br>` as decorative) is **unwrapped**: the paragraph spacing (`\n\n` leading and trailing) is dropped and the children render as if `<p>` weren't there. Covers Confluence-editor "cursor-park wrap" patterns where the author typed a single inline element (`<p><code>X</code></p>`, `<p><span style="color:red;">X</span></p>`, `<p><strong>X</strong></p>`, `<p><a href="…">X</a></p>`) on its own line and the editor wrapped it in `<p>`. The unwrap is structural — surrounding `<p>`s (the ones with real text content) still emit their own `\n\n`, so an unwrapped `<p>` between two text paragraphs glues to the preceding paragraph but the following paragraph still starts on a new line (asymmetric on purpose — the unwrap rule only suppresses the wrapping `<p>`'s OWN spacing, not the siblings'). Subsumes the older "macro-only `<p>` is unwrapped" carve-out (an `<ac:structured-macro>` is a Tag, so the broader rule fires too).
- **Line breaks**: `<br/>` becomes a plain `\n` — no trailing backslash. Visual line break is preserved in Obsidian's reading view under the default **"Strict line breaks" = OFF** setting, which converts a soft break (single source newline inside a paragraph) into a rendered `<br>`. Under strict-line-breaks ON, soft breaks render as spaces and `<br/>`-derived line breaks would be lost — not supported, same assumption as the multi-`<code>` `<pre>` rule. Was previously a backslash hard break (`\` at end of line) which forced the `<br>` under both settings; dropped because the visible trailing `\` made source `.md` files noisy and some editors / linters strip trailing whitespace adjacent to it. Inside a `<pre>` block, `<br/>` is also `\n` (same handler) and feeds the multi-`<code>` line splitter directly.
- **List nesting indent**: nested lists are indented with one tab character per level (not spaces).
- **Adjacent ordered-list break**: when two ordered lists end up back-to-back in the rendered markdown — same indent depth, no separating block between them — a single-line empty blockquote (`> ` at the matching indent, surrounded by blank lines) is inserted between them so they render as two distinct lists rather than one merged sequence. Trigger: at any indent depth `N`, an ordered list line (`\d+\. …`) immediately followed (one or more newlines later) by a line that starts with `1\. ` at the same depth — the `1` is the tell-tale "new list restarts numbering" signal, since the converter always emits each `<ol>`'s items starting at 1. Injection: replace the `\n+` gap with `\n\n<indent>> \n\n` so the separator's indent matches the surrounding lists, keeping it nested-safe. Why an empty blockquote: CommonMark merges consecutive ordered lists into one (renumbering 1, 2, 3, … across the merge) and a blank line alone doesn't break them; a different block element between them does, and `> ` is the lightest visible option (single character + space). Realistic source: in this corpus the trigger only fires when Confluence emits a cursor-park `<p><br/></p>` between two `<ol>` siblings — the cursor-park is dropped to empty (per the `<p>` rule above), the two lists land adjacent in the output, and the post-process injects the breaker. Detection lives at the output boundary (post-process) rather than at the `<p>` site so the rule catches the bad adjacency regardless of what source pattern produced it.
- **List depth-jump normalization**: after the converter produces the markdown output, a final post-process pass scans for list lines whose indent depth exceeds the natural progression and injects empty placeholder list items at every missing depth so the indentation chain is unbroken from depth 0 to the deepest line. A "list line" is any line matching `^\t*- ` or `^\t*\d+\. `; its **depth** is the number of leading tabs. The pass maintains a running stack of "live" depths as it walks line-by-line: a list line at depth N must have parents at every depth `0..N-1` already in the stack — if any are missing, a bare `- ` placeholder is injected at each missing depth before the line is emitted, then the stack is trimmed to `0..N`. Any non-list line (plain text, blank line, heading, callout, code fence, table HTML, etc.) **resets the stack** — the next list line starts a fresh hierarchy. Placeholder marker is **always `-`**, never `1.`, regardless of the deeper line's style: ordered lists with depth jumps get bulleted placeholders (`1. foo` at depth 2 with no parents → `- \n\t- \n\t\t1. foo`) — the placeholder has no content so the "1" numbering would be meaningless, and a single placeholder style is easier to scan visually. The deeper line's marker is preserved verbatim (no renumbering, no ordered→bulleted conversion). The transformation is **idempotent**: applying it to already-valid markdown produces no change. Defensive: catches depth-jump cases regardless of which Confluence source pattern triggered them (rare in practice, but the renderer's recursion plus block-hoisting interaction makes the invariant worth enforcing at the output boundary rather than trusting every path through `_render_list_inner` to preserve it).
- **`<p>` inside `<li>`**: the first `<p>` child of an `<li>` **whose rendered content is non-empty** is treated as a paragraph-break **split point**: its inline content becomes the list item's inline body, and every sibling that appears *after* that `<p>` (whether `<p>`, raw text, inline tag, or block-emitting tag) is **hoisted** out and rendered at column 0 on the line immediately following the list item's inline content (same hoist mechanism as the existing "block content inside `<li>`" rule). Without this rule the post-`<p>` siblings glue onto the list item line with no separator: `<li><p>hello world</p>after-text</li>` renders as `- hello worldafter-text` and `<li><p>X</p><strong>Y</strong></li>` renders as `- X**Y**`. The cause is that `<p>` renders to `\n\nX` whose stripped form has no embedded newline, so the existing block-content hoist (which keys on stripped output containing `\n`) doesn't fire — the `<p>` body and surrounding siblings all land in `inline_parts`, and the body builder strips leading whitespace, collapsing the separation. The `<p>`-split rule fires for any `<p>` child regardless of class — `auto-cursor-target` (Confluence's auto-generated-paragraph marker) is the most common but not the only trigger. **Empty `<p>`s** (cursor-park `<p></p>` / `<p><br/></p>` Confluence inserts around real content) are skipped — they don't claim the split, so the next sibling is free to land as inline content on the list-item line. Without this carve-out, `<li><p><br/></p><latex-inline/></li>` would render as a bare `- ` line followed by a hoisted `$X$` on the next line; with it, the latex stays inline on the marker line: `- $X$`. Content *before* the first non-empty `<p>` (rare in practice — Confluence's editor wraps typed text in `<p>` automatically) joins the inline body via existing inline_parts concatenation.
- **Block content inside `<li>`**: when a list item contains a child whose rendered output spans multiple lines (`expand` / `ui-expand` / `info` / `warning` callouts, fenced code blocks, multi-line `<table>` HTML, etc., detected as any rendered child whose stripped output contains a `\n`), the block is hoisted out of the list item and emitted at column 0 on the line immediately following the list item's inline content. Adjacent hoisted blocks are separated by a blank line so Obsidian renders them as separate blocks. The list is implicitly terminated at column 0 — the hoisted block renders as a top-level element, not as visual content of the list item. Each occurrence is logged to `migration-report.md` under `## Warnings` with the page title — the original Confluence relationship ("this block belongs to this list item") is lost in conversion and the page may need manual review. The list marker line **always keeps its trailing space** after the marker (`- ` for bullets, `1. ` for ordered) even when the list item's inline content is empty — CommonMark requires a space after the marker for it to be recognized as a list bullet (a bare `-` on its own line parses as plain text, not a list item), so the empty-body-followed-by-hoisted-block case (e.g. an `<li>` containing only an `expand` macro) emits `- ` on the marker line and the hoisted callout on the next line, preserving the visual "empty bullet + block" intent.
- **Inline HTML preserved**: `<sub>` and `<sup>` are emitted as raw HTML (`<sub>2</sub>`, `<sup>2</sup>`) — Markdown has no native equivalent and Obsidian renders them in reading view. Same approach as the existing `<u>` handling. `<strong>` / `<b>` and `<em>` / `<i>` are **also** emitted as raw HTML (`<strong>foo</strong>`, `<em>foo</em>`) rather than as their Markdown equivalents (`**foo**`, `*foo*`) — Markdown bold/italic are fragile under several common patterns (e.g., `<strong>X</strong><strong>Y</strong>` becomes `**X****Y**` which Markdown parses ambiguously; an isolated `<strong> </strong>` becomes `** **` which renders as literal asterisks with a space). Raw HTML survives all those cases cleanly because Obsidian renders the tags directly. **Whitespace-only carve-out**: when the inner rendered content of a `<strong>` / `<b>` / `<em>` / `<i>` is whitespace-only (or absent entirely), the tag is dropped and the inner whitespace emitted directly — so `<strong> </strong>` becomes a single space, `<strong></strong>` becomes nothing, and nested `<strong><em> </em></strong>` collapses recursively to a single space. Confluence's editor frequently emits these empty-formatting tags when a user toggles bold/italic while their cursor is over a space character; the formatting carries no semantic content, and preserving the wrapper produces visual noise in the output. The carve-out does NOT apply to `<u>` / `<sub>` / `<sup>` — those are kept verbatim even when whitespace-only (rare, and they're already much less common in Confluence content).
- **Plain-text Markdown escape**: in user-visible body text, the characters `\`, `` ` ``, `*`, `_`, `~`, `#`, `[`, `]`, `$`, `<`, and `>` are each prefixed with a backslash (e.g. `\` → `\\`, `` ` `` → `` \` ``, `*` → `\*`, `_` → `\_`, `~` → `\~`, `#` → `\#`, `[` → `\[`, `]` → `\]`, `$` → `\$`, `<` → `\<`, `>` → `\>`) so Obsidian renders them as literal characters instead of triggering Markdown syntax (inline code / bold / italic / strikethrough / tags / links / footnotes / task lists / math / inline HTML). The escape over-covers a few low-risk positional cases — intraword `_` and mid-line `#` are technically safe in CommonMark, but are escaped anyway for simplicity and to defeat Obsidian's tag extension (`#123` would otherwise become a tag). Source `.md` files will show backslash-prefixed characters more often as a result; rendered output is unchanged. Applied to text nodes whose ancestor chain includes any of `<p>`, `<h1>`–`<h6>`, `<li>`, `<strong>`, `<em>`, `<u>`, `<sub>`, `<sup>`, or an uncolored `<span>` (i.e. a `<span>` whose `style` does not contain `color:` — colored spans become raw `<font>` HTML, where Obsidian does not process Markdown escapes, so their inner text is left verbatim). The escape is **suppressed** whenever a closer ancestor is `ac:plain-text-body` or `ac:plain-text-link-body` (CDATA bodies that become fenced code blocks or latex math — no HTML parsing happens inside those constructs, so `<>` stay literal). Inside `<code>` and `<pre>` (HTML inline-code and the multi-`<code>`-rendered pre block), the escape is **narrowed to only `<` and `>`** — the other Markdown metacharacters render literally inside `<code>` anyway (Obsidian doesn't process Markdown inside inline HTML), but `<>` are still parsed as HTML-tag boundaries by Obsidian's HTML parser, so they need escaping to render as literal characters. Inside `<a>` and `<ac:link>` (link display text), only `[`, `]`, `<`, and `>` are escaped — the other characters stay verbatim so inline formatting and math survive inside the link (e.g. `[**bold link**](url)` and `[$x^2$](url)` still render correctly), while the brackets escape so a user-typed `[draft]` inside link text doesn't break the constructed `[text](url)` parse into `[[draft]](url)` and the angle brackets escape so a user-typed `<tag>` in link text doesn't get parsed as inline HTML. Table-cell text follows the separate Tables rule below (no plain-text escape applied). Already-escaped sequences are not detected: a Confluence-typed literal `\]` becomes `\\\]` (four characters) in the output, which Obsidian renders back to `\]` — preserving the original two-character text.
- **Inline `<code>`**: always emitted as raw HTML `<code>X</code>` (not Markdown backticks). Keeps rendering consistent with colored inline code below.
- **Block `<pre>`**: not a code block — emitted as a sequence of inline `<code>` elements, one per rendered line, **joined by literal `<br>` HTML tags** (e.g. `<code>line1</code><br><code>line2</code><br><code>line3</code>` on a single source line). Confluence syntax-highlighted snippets carry colors via nested `<span style="color:...">` children which die inside a Markdown fenced code block (Markdown doesn't process HTML inside `` ``` ``); multi-`<code>` lets the existing colored-span → `<font>` rule compose so highlight colors survive. Lines come from splitting the rendered child content on `\n` **and** `<br/>` (both produce line breaks identically — and both then become the same `<br>` output separator between `<code>` segments). Leading and trailing empty lines are trimmed; internal empty lines are preserved as empty `<code></code>` placeholders surrounded by `<br>` separators (vertical spacing inside the snippet matters — e.g. paragraph breaks in HTTP transcripts, log dumps). An empty `<pre>` (no content, or only whitespace) emits nothing. Each line's children render through the **full document-mode Markdown converter** — colored-span → `<font>` and uncolored-span unwrap compose naturally; `<u>` / `<sub>` / `<sup>` survive as raw HTML (Obsidian renders HTML inside inline `<code>`); but Markdown-output tags (`<strong>` → `<strong>foo</strong>`, `<em>` → `<em>foo</em>`, `<a>` → `[text](u)`) — the first two work because Obsidian renders inline HTML inside `<code>`; the `<a>` Markdown link does NOT render inside `<code>` and surfaces as a literal `[text](u)` string. Rendering is **independent of Obsidian's "Strict line breaks" setting** — the literal `<br>` is HTML, parsed regardless of source-newline interpretation. The rule applies **at both document level and inside table cells**: a `<pre>` inside a `<td>` / `<th>` gets the same multi-`<code>`-with-`<br>` rendering, no warning logged, no raw `<pre>` pass-through (older behavior). (Was previously a fenced code block — switched to multi-`<code>` to preserve syntax-highlighting colors that fenced blocks strip. Then the separator was `\n`, relying on Obsidian's default strict-line-breaks=OFF to convert soft breaks to `<br>`; switched to literal `<br>` so the rule works under either setting and to enable unified document/cell behavior — `<br>` is valid inside `<td>` while a source newline isn't.)
- **Text color**: a `<span>` whose `style` attribute contains `color:` becomes `<font style="<original style>">X</font>` — the entire `style` attribute is copied verbatim, and the inner content `X` is rendered **verbatim with leading/trailing whitespace preserved** (no `.strip()`). This matters for Confluence's syntax-highlighted code blocks, which wrap each token in its own colored span — a trailing space inside `<span style="...">refresh </span>` MUST survive so adjacent tokens don't glue together in the rendered `<font>`-sequence (`refresh </font><font>-</font>` not `refresh</font><font>-</font>`). **Block-content unwrap**: when the colored span's rendered content is **block-shaped** — i.e. the stripped output contains an embedded `\n`, indicating a multi-line construct like a dataview block (from a `children` / `recently-updated` macro), a fenced code block (from a `code` macro), a callout (from `info` / `warning` / `expand`), or any other block-emitting child — the `<font>` wrapper is **dropped** and just the inner content is emitted. The `<font>` tag can't meaningfully color a block construct anyway, and the wrapper survives in the output as inert noise that visibly clutters the source `.md`. Nested colored spans unwrap recursively for free: an inner `<span color>` wrapping the macro drops its wrapper, the outer `<span color>` then sees multi-line content from the inner and drops its wrapper too. Inline (single-line) content is unaffected — `<span color>text</span>` still becomes `<font>text</font>`. Matches the Obsidian font-color plugin convention. Spans without a `color:` declaration keep the existing strip-wrapper behavior (children rendered, no wrapper). Composes with the inline-`<code>` rule above when the source is `<code><span style="color: rgb(...);">X</span></code>` → `<code><font style="color: rgb(...);">X</font></code>` (the natural nesting carries through). The **reverse source nesting** — `<span style="color: rgb(...);"><code>X</code></span>` — is normalized to the same code-outer shape: when a colored span's children consist of **exactly one `<code>` element** (ignoring purely-whitespace text nodes), the span/code pair is rewritten as `<code><font style="color: rgb(...);">X</font></code>` — `<code>` ends up outermost regardless of which order Confluence's editor emitted. Obsidian's font-color plugin only colors text reliably when the `<font>` sits *inside* the monospace `<code>` block; the other nesting (`<font><code>X</code></font>`) leaves the code text in its default colour because `<code>`'s monospace styling supersedes the surrounding `<font>` color. The normalization is **narrow**: it only fires for sole-`<code>`-child spans. A colored span with mixed content (`<span style="">prefix <code>X</code> suffix</span>`) or multiple `<code>` children stays as `<font>...</font>` wrapping the whole thing — restructuring those would re-parent siblings and can break adjacent text layout, and the case is unusual in Confluence content.
- **Macro: Children Display** → inserted only when the Confluence page explicitly contains the macro; renders as a Dataview LIST block wrapped in one or two Obsidian callouts that simulate list-indentation alignment (requires custom CSS in the destination vault on `[data-callout="list-indent-undo"]` and `[data-callout="indent"]` to produce the visual effect — without the CSS, the callouts render as default gray boxes). Two forms for the dataview scope:
  - **No `page` parameter**: scoped to this page's children — `WHERE file.folder = this.file.folder + "/" + this.file.name`
  - **With `page` parameter** (e.g., `<ri:page ri:content-title="Computer"/>`): scoped to the referenced page's children — `WHERE file.folder = [[Computer]].file.folder + "/" + [[Computer]].file.name`. The wiki-link uses the sanitized/collision-resolved title (same resolver as internal page links). If the referenced page is not in the migrated title map, the macro is dropped and a broken-reference entry is added to `migration-report.md`.
  - **Callout wrapping** (independent of the scope above): if the `<ac:structured-macro ac:name="children">` element has **no `<li>` ancestor** in the source HTML (it appears at document level, outside any list), the dataview block is wrapped in a single `[!list-indent-undo]` callout: `> [!list-indent-undo]\n> ```dataview\n> LIST\n> FROM ""\n> WHERE …\n> ``` `. If the element **has any `<li>` ancestor** (at any depth, including nested lists), it's wrapped in a doubly-nested callout: `> [!list-indent-undo]\n> > [!indent]\n> > ```dataview\n> > LIST\n…\n> > ``` `. Both forms emit a trailing blank line after the closing fence.
  - **Empty-`<li>` marker drop**: when a `<li>`'s rendered inline content is empty and the item has hoisted block content (multi-line block — `expand` / `info` / `warning` / `code` callouts, a Children Display callout, an image-in-`<li>` callout, etc.), the list-item's `- ` / `1. ` marker line is **dropped entirely** — the hoisted block is the sole output for that `<li>`. Without this drop, the output would carry an orphan `- ` line above the block. The rule is **general** (applies to any empty-body `<li>` with block content), not specific to Children macros — image-only `<li>`s and `expand`-only `<li>`s drop their markers the same way.
- **Heading levels**: `<h1>` → `#`, `<h2>` → `##`, `<h3>` → `###`; `<h4>`, `<h5>`, `<h6>` all clamp to `######` (the deepest Markdown heading)
- **Macro: Excerpt** → rendered Markdown body wrapped in a fenced `` ```excerpt `` code block followed immediately by a `^excerpt` block anchor on the next line; rendered by a custom Obsidian plugin
- **Macro: Include Excerpt** → `![[Page Title#^excerpt]]` Obsidian block transclusion; page title extracted from `ac:parameter[ac:name=""]` → `ri:page[@ri:content-title]`; `nopanel` parameter ignored
- **Macro: `latex-inline`** / **`latex-block`** / **`latex`** → `$<CDATA>$` (Obsidian inline math). All three variants produce the same `$X$` math wrapping — `latex-block` is treated as an alias for `latex-inline`, not as Obsidian block math (`$$X$$`). The only difference is positional: `latex-block` emits a trailing `\n` after the closing `$` (output `$X$\n`) so consecutive `latex-block` macros land on separate source lines instead of gluing together; `latex-inline` emits no trailing newline (output `$X$`) so it flows within prose. Body normalization runs in two steps: (1) any run of newlines in the CDATA body (together with whitespace on either side) is collapsed to a **single ASCII space** — adjacent lines are joined with one space between them (Obsidian's inline math renderer breaks on embedded newlines; LaTeX is whitespace-insensitive between tokens, so `\sum\nx` → `\sum x` renders identically to the source and a multi-line `\text{some\ntext}` correctly becomes `\text{some text}`); (2) every remaining whitespace character (`\s` — Unicode whitespace such as NBSP `U+00A0`, em-space `U+2003`, narrow no-break space `U+202F`, etc., plus stray ASCII tabs / vertical tabs / form feeds) is replaced **one-to-one** with an ASCII space `U+0020`. Confluence's editor frequently inserts NBSPs where the author typed an ordinary space (Shift+Space, paste-from-formatted-source), and these visibly differ — or fail to parse — in Obsidian's math renderer. Same broad `\s`-to-space normalization is used for `ac:image` filenames (see Tables). One edge case where space-separator is technically wrong: a Confluence author who line-breaks immediately before a subscript / superscript marker (`\sum\n_{i=1}^n`) gets a separating space (`\sum _{i=1}^n`), which LaTeX may parse as `\sum` followed by an independent `_{...}` rather than as `\sum`'s subscript — accepted because the source corpus contains no multi-line equations of this shape. (Was previously split: `latex-block` rendered as `$$X$$` with the multiline body preserved verbatim. Collapsed into `latex-inline` because every `latex-block` instance in the source corpus contains a single-line equation — the block-vs-inline distinction added no value and forfeited inline positioning within prose. Re-introduce the split if the corpus ever grows multi-line equations: matrices via `\begin{matrix}…\end{matrix}`, aligned environments via `\begin{aligned}…\end{aligned}`, etc.) **Every `latex-block` occurrence is logged to `migration-report.md` with the page title** — one warning per occurrence, both at document level and inside table cells, so the previously-block-math content can be spot-checked in Obsidian. `latex-inline` and the rare `latex` alias are NOT logged (the visible-rendering change only affects `latex-block`).
- **Macro: `code`** → fenced code block with language from `ac:parameter[language]`
- **Macro: `info`** → `> [!info]` callout wrapping the body content
- **Macro: `warning`** → `> [!warning]` callout wrapping the body content
- **Macro: `expand`** / **`ui-expand`** → native Obsidian callout, foldable and collapsed by default: `> [!expand]- TITLE` (input macro `expand`) or `> [!expand-ui]- TITLE` (input macro `ui-expand`), with the body on subsequent `> `-prefixed lines. Body is rendered through the **full document-mode Markdown converter** — lists, code blocks, headings, callouts, internal links, nested tables, nested expands all render natively (Obsidian re-parses callout contents as Markdown) — then stripped, 3+ consecutive newlines collapsed to 2, every line prefixed with `> ` (empty lines become bare `>`). Nesting falls out for free — an inner expand's own `> ` prefixes become `> >` under the outer one. When `title` is absent, the callout header defaults to `Click here to expand...` for both macros. Empty body still emits the header line (never dropped). Note the asymmetric output naming: input macro `ui-expand` → output callout type `expand-ui`. Visual differentiation between `expand` and `expand-ui` depends on **custom CSS** on `[data-callout="expand"]` and `[data-callout="expand-ui"]` in the destination vault — Obsidian has no built-in styling for these callout type names; without the CSS, both render as plain "note" callouts. Cell-context expand keeps the inline `<details><summary>…</summary>…</details>` form (cells can't host block-level callouts) — see the **Tables** decision. (Was previously fenced blocks for a custom Obsidian plugin — switched to native callouts to drop the plugin dependency; callouts re-parse Markdown internally the same way the plugin did.)
- **Macro: `anchor`** → silently dropped (Obsidian has no equivalent free-floating anchor; not logged as unknown)
- **Macro: `toc`** → silently dropped (Obsidian's built-in Outline pane renders the heading tree natively)
- **Macro: `view-file`** / **`viewpdf`** / **`multimedia`** → `![[PageName/filename]]` where filename comes from the embedded `ri:attachment`; the attachment is downloaded into the per-page subfolder (same flow as inline images); Obsidian renders PDFs, video, and audio inline natively in reading view
- **Macro: `ui-tabs`** → fenced block for the **["Tabs" Obsidian plugin](https://github.com/ycnmhd/obsidian-tabs)**, with the literal opening fence `~~~tabs`, one `---tab <TITLE>` separator per child `ui-tab`, and the literal closing fence `~~~`. Preserves horizontal-tab semantics in Obsidian's reading view. Assembled by the `ui-tabs` renderer itself (not by delegating to per-tab rendering): it scans direct `ui-tab` descendants of its `ac:rich-text-body` in document order, rendering each tab's `title` parameter on a `---tab TITLE` line and the tab's `ac:rich-text-body` immediately below (no blank line between). Tab body is rendered through the **full document-mode Markdown converter** — lists, code blocks, callouts, internal links, nested tables, nested expands all work natively inside a tab; each body is stripped of leading/trailing whitespace. Non-`ui-tab` content inside the `ui-tabs` body is dropped silently (matches Confluence's visual behaviour — its tab UI ignores stray content too; cursor-parking `<p><br/></p>` paragraphs are the common case). A `ui-tabs` with zero `ui-tab` children produces no output (the whole block is dropped). `ui-tab` is **not** dispatched by `_render_macro` — it appears only as a child of `ui-tabs` and is rendered inline by the parent; an orphan `ui-tab` outside a `ui-tabs` is an invariant violation and would surface as an unknown-macro entry in `migration-report.md`. (Was previously stacked `<details>` collapsibles — switched to the plugin format to recover horizontal tab layout.)
- **Macro: `widget`** → `![](url)` Media Extended embed; URL extracted from `ac:parameter[ac:name="url"]` → `ri:url[@ri:value]`; used in practice only for YouTube embeds
- **Macro: `section`** → entire macro and body dropped (no output, not logged as unknown); used for multi-column layouts that have no Markdown equivalent
- **Macro: `details`** → entire macro and body dropped (no output, not logged as unknown); Confluence Page Properties metadata has no Markdown equivalent
- **Macro: `pagetree`** / **`pagetreesearch`** / **`livesearch`** → silently dropped; Obsidian's file explorer and global search cover these natively
- **Macro: `recently-updated`** → Dataview query `LIST FROM "" SORT modified DESC LIMIT <max>` where `<max>` comes from the `max` parameter (default `15`); sorts by the `modified` frontmatter field (Confluence-recorded timestamp); `spaces`, `types`, and `theme` parameters are ignored
- **Internal page link** → `[[Page Title|Display Text]]` Obsidian wiki link (page titles are unique across the instance)
- **External URL link** → `[Display Text](url)` standard Markdown link
- **YouTube embed** (`<ri:url>`) → `![](url)` Media Extended plugin syntax
- **Attachments**: downloaded into per-page subfolder `<Space>/<Parent>/<PageName>/<filename>`, referenced as `![[PageName/filename]]`; inline images embedded in page body (`<ac:image><ri:attachment>`) are treated identically — downloaded to the same subfolder and emitted as `![[PageName/filename]]` at the point of the image. Inline-image sizes carry over via the **Obsidian wiki-link size suffix** only when the source `<ac:image>` carries an `ac:width` attribute: `ac:width="W"` alone → `![[PageName/filename|W]]` (Obsidian scales height proportionally); `ac:width="W"` plus `ac:height="H"` → `![[PageName/filename|WxH]]`; neither attribute (or `ac:height` alone) → bare `![[PageName/filename]]`. `ac:height` is intentionally **ignored when `ac:width` is absent** — Obsidian's `|0xH` "height-only" convention does not render reliably in practice (image either renders unsized or with the wrong aspect ratio), so the safer fallback is to drop the size hint and let Obsidian display the image at its natural size. Values are read verbatim from the attributes — no parsing, casting, or unit-stripping (Confluence's documented format is integer pixels; any non-integer value passes through and may render wrong, but that mirrors the cell `<img>` rule below). The `ac:style` attribute (e.g. `max-height: 250.0px;`) is **never** parsed — Confluence's editor sets `ac:width` / `ac:height` alongside any inline style, so attribute-only reading is sufficient in practice. External-URL images (`<ac:image><ri:url>`) stay as `![](url)` with no size — Markdown image syntax has no native size suffix. **Image inside `<li>`** (any depth) gets the same callout-wrap treatment as a Children Display macro inside `<li>`: the `![[Page/file|W]]` wiki-link is wrapped in a doubly-nested `> [!list-indent-undo]\n> > [!indent]\n> > ![[Page/file|W]]` callout and hoisted to column 0 below the list item, so it visually aligns flush with the list bullet (requires the same custom CSS on `[data-callout="list-indent-undo"]` and `[data-callout="indent"]` as the children-macro rule). Images at document level (no `<li>` ancestor) emit unchanged — bare `![[Page/file|W]]` with no callout wrap, since there's no list indentation to align against. The 2-wrap form fires only for the document-level `<ac:image>` path: table-cell `<ac:image>` (which becomes `<img src="…" />` in the cell HTML) and external-URL images are unaffected. Any Unicode whitespace in the filename is normalized **one-to-one** to ASCII space `U+0020` (same `\s`-to-space pattern used for `ac:image` in table cells and for latex bodies) at **both** sites that touch the filename: every wiki-link reference emitted by the converter, **and** the on-disk filename when the attachment is written. Both must agree — Confluence's editor sometimes embeds NBSP (`U+00A0`) etc. in attachment filenames (most common: screenshots whose default names contain NBSP-separated timestamps), and without paired normalization Obsidian can't resolve the link to the file on disk. Applies to all reference paths: inline `<ac:image>`, attachment `<ac:link>`, and the `view-file` / `viewpdf` / `multimedia` macros.
- **Unknown macros**: logged and skipped; a `migration-report.md` written to vault root with macro name, page, and count
- **Tables**: every `<table>` is emitted as cleaned, prettified raw HTML (no Markdown pipe tables — Obsidian doesn't render Markdown inside `<table>`). Cleaning: strip `<colgroup>` and its contents; on every element, drop all attributes except `colspan`, `rowspan`, `data-highlight-colour`, and `style` (kept verbatim); on `<td>`/`<th>`, `data-highlight-colour="X"` is appended to the cell's `style` as `background-color: X;` (merging with any existing `style`); the colour name `grey` is remapped to `#F4F5F7`. Every `<th>` with no explicit background-color (no `data-highlight-colour`, no `background-color:` declaration in `style`) receives an automatic `background-color: #F4F5F7;` (Confluence's default header tint). Cell contents stay as verbatim Confluence HTML with these exceptions: `<ac:link>` is replaced by plain text (the `<ac:plain-text-link-body>` CDATA if present, else the `ri:page ri:content-title` value); `<ac:structured-macro ac:name="expand">` / `ac:name="ui-expand"` is converted to an inline single-line `<details><summary>TITLE</summary>CONTENT</details>` (no blank lines, no newlines between elements) — `TITLE` from `ac:parameter[ac:name="title"]` falling back to `Click here to expand...` for `expand` when absent; `CONTENT` is the `ac:rich-text-body` rendered recursively under these same table-cell rules (so nested `ac:*` keeps being escaped, nested `ac:link`/`ac:image`/`expand`/`ui-expand` apply their cell exceptions, nested expands stay inline single-line); `<ac:structured-macro ac:name="latex-inline">` / `ac:name="latex-block"` / `ac:name="latex"` is converted to a single-line `<span class="math display">BODY</span>` element — the same `<span class="math display">` wrapper is used for both inline and block latex variants in cells (Obsidian's math plugin reads the `math display` class to render the inner text as LaTeX, and the cell context doesn't support distinct inline-vs-block math layouts anyway). `BODY` is the CDATA from `<ac:plain-text-body>`, stripped and run through the same two-step normalization as the document-level latex handler: any run of newlines (with surrounding whitespace) collapses to a single ASCII space, then any remaining `\s` character (NBSP `U+00A0`, em-space, etc.) is replaced one-to-one with ASCII space. After normalization, `<`, `>`, and `&` in `BODY` are HTML-escaped so the surrounding `<span>` stays valid HTML (LaTeX commonly uses `<` and `>` as inequalities or as `\langle\rangle` arguments). Every other `<ac:*>` element is serialized back to its source XML and HTML-escaped (`<` → `&lt;`, `>` → `&gt;`) so it appears as visible literal text in the rendered cell — a manual-cleanup signal — **and a warning is logged to `migration-report.md` for every such fall-through**: one entry per occurrence with the page title and the element identifier (for `<ac:structured-macro>` the `ac:name` is included; for other `<ac:*>` elements the tag name itself). The log covers any `ac:*` that lands in the escape branch, including known macros that have no cell transform (`info`, `warning`, `code`, `excerpt`, `excerpt-include`, `children`, `widget`, `ui-tabs`, `recently-updated`, `section`, `details`, `anchor`, `toc`, `pagetree`, etc.) and any genuinely unknown macro — every one of them surfaces as visible XML in the rendered cell, so the report gives the operator a complete cleanup checklist. One further exception: `<ac:image>` containing a `<ri:attachment ri:filename="X">` is converted to `<img src="X" />` (bare filename — no per-page subfolder prefix; may need manual fixup; any Unicode whitespace in the filename is normalized to ASCII space), and `ac:width` / `ac:height` (if present) carry over as `width` / `height` attributes; all other `ac:image` attributes are dropped. An `<ac:image>` lacking a usable `ri:filename` falls back to the escape-as-source-XML treatment and is logged to `migration-report.md` with the page title. Three wrapper-stripping rules apply inside cells: `<div class="content-wrapper">` is always unwrapped (Confluence noise); a `<p>` whose direct children are **Tag-only** (no direct text-node content, with whitespace-only text and `<br>` treated as decorative) is unwrapped — same `_is_tag_only_paragraph` rule used at document level, so the Confluence-editor cursor-park pattern of wrapping a single inline element in `<p>` (`<p><code>X</code></p>`, `<p><span style="color:…">X</span></p>`, `<p><ac:link/></p>`, `<p><ac:image/></p>`, etc.) flows through as just the inline element inside the cell with no `<p>...</p>` HTML noise; and a `<p>` whose only non-whitespace content is one or more `<br/>` tags is **dropped entirely** (Confluence emits these as cursor-parking spots — e.g. `<p class="auto-cursor-target"><br /></p>` — and they would otherwise leave visible vertical noise in the cell). Attributes on the `<p>` are ignored when matching this rule. Prettification: 4-space indent, one element per line — except inside `<td>`/`<th>`, where children render inline; raw text nodes inside a `<td>` that already contain literal newlines preserve them. When a `<table>` appears inside another table's cell, the inner table is recursively rendered with the same prettification rules, indented at parent-cell-depth + 1, **but with no `\n` before the opening `<table>` or after the closing `</table>`** — the nested table sits flush against any preceding/following content in the parent cell (`BEFORE<table>...</table>AFTER`). The inner table keeps its internal multi-line structure for rows and cells; only the boundary newlines against the parent cell are removed. Asymmetric: the opening `<table>` has no leading indent (it's inline), but the closing `</table>` keeps its depth-aligned indent — visible only in source HTML, irrelevant to Obsidian's renderer. Followed by a blank line per the block-spacing rule (except when nested inside another cell — those stay flush as described above).
- **Filename sanitization**: characters illegal in filenames (`:`, `/`, `?`, `*`, `<`, `>`, `|`, `\`) in page titles are replaced with their visually similar Unicode fullwidth equivalents (`：／？＊＜＞｜＼`) when used as file or directory names — preserves the appearance and is lossless (no two distinct Confluence titles collapse to the same on-disk name); post-sanitization collisions are still resolved by suffixing the Confluence page ID and logged in `migration-report.md`
- **Migration mode**: one-time, full re-run on every invocation (no incremental sync, no resume); Ctrl+C triggers a graceful stop that writes `migration-report.md` before exiting
- **migration-report.md**: written to vault root on graceful stop, crash, or completion; contains all unknown macros encountered (macro name, page title, count) and all per-page failures (page title, error reason)
- **Frontmatter**: created, modified, labels (as Obsidian tags), **parent** (omitted on root pages with no ancestors), **children** (always emitted — empty list `[]` on leaf pages). Both `parent` and `children` use the **sanitized on-disk basename wrapped in `[[…]]`** (Obsidian wiki-link form) — e.g. `parent: "[[Architecture]]"`, `children: - "[[Module A]]"`. The wiki-link wrapping makes the property values clickable in Obsidian's reading view and surfaces them to Dataview as link objects (rather than opaque strings). The basename inside the brackets is the same form that appears in the `<title>.md` filename — **without** the `.md` extension — matching Obsidian's wiki-link resolution. YAML-quoted as `"[[X]]"` (unquoted `[[X]]` would be invalid YAML — `[` starts a flow sequence). Empty `children` stays as `children: []` (nothing to wrap). The parent → children relationship is derived once during migration from the `ancestors` array of every page in the page set; each page's `children` is the set of OTHER pages whose direct parent (`ancestors[-1]`) is this page (kept stable across runs via alphabetical sort). Pages outside the current migration scope (e.g., a parent in a space that wasn't migrated, or children excluded by the space-key filter) are NOT followed — `parent` may be omitted even on a non-root page if its parent isn't in the migration set, and `children` only includes pages that were actually migrated.
- **Config**: `.env` file (`CONFLUENCE_URL`, `CONFLUENCE_USER`, `CONFLUENCE_PASS`, `VAULT_PATH`, `CONFLUENCE_VERIFY_SSL`, `DOWNLOAD_ATTACHMENTS`); space keys passed as CLI args
- **Progress display**: each page migration prints `[<n>/<total>] <Page Title>` to stdout
- **SSL**: verification skipped (`verify=False`), warnings suppressed; controlled by `CONFLUENCE_VERIFY_SSL` env var
- **Attachment download**: controlled by `DOWNLOAD_ATTACHMENTS` env var (default `true`); when `false`, attachments are skipped and attachment references are emitted as broken wiki-links (e.g., `![[PageName/diagram.png]]`)
