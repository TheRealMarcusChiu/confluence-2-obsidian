# confluence-2-obsidian Context

## Glossary

### Source
Confluence Cloud or Server instance, accessed via REST API.

### Sink
Obsidian vault тАФ a local directory of Markdown files.

## Decisions

- **Data source**: Confluence Server REST API (self-hosted, v7.13.0, https://confluence.lan/)
- **API base**: `https://confluence.lan/rest/api/`
- **Auth**: Basic Auth (username + password in config/env)
- **Scope**: specific spaces, identified by space key, passed as CLI arguments
- **Content types**: pages and attachments only (no blog posts, no comments)
- **Vault structure**: mirrors Confluence page tree at full depth. Every page is emitted as both a `.md` file AND a same-named directory at the same level (e.g., `PROJ/Architecture.md` alongside `PROJ/Architecture/`). The directory holds the page's child pages (recursively) and any per-page attachments. Leaf pages with no children and no attachments still get an empty directory тАФ designed for the Obsidian "Folder Notes" plugin, which treats the sibling `.md` as the folder index.
- **Language**: Python
- **HTML-to-Markdown**: custom recursive parser using BeautifulSoup; walks the Confluence XML tree in one pass, handling standard HTML elements and `ac:` macros together
- **Block spacing**: tight packing тАФ no blank lines between block-level elements (headings, code blocks, lists, callouts, dataview blocks, tables, etc.). Three exceptions: (1) consecutive paragraphs are separated by a blank line (Markdown requires it), (2) `^excerpt` block anchor is always followed by a blank line, and (3) every `</table>` is followed by a blank line тАФ required for the HTML block to terminate so subsequent Markdown parses (CommonMark ┬з4.6). Suppressed for tables nested inside another `<td>`/`<th>` (must sit flush per the Tables rule). A `<p>` containing only a structured-macro is unwrapped (the macro renders as its own block, no paragraph spacing)
- **Hard line breaks**: `<br/>` outside HTML contexts тЖТ backslash hard break (`\` at end of line); `<br/>` inside raw HTML (e.g., within a color span) needs no special handling
- **List nesting indent**: nested lists are indented with one tab character per level (not spaces).
- **Block content inside `<li>`**: when a list item contains a child whose rendered output spans multiple lines (`expand` / `ui-expand` / `info` / `warning` callouts, fenced code blocks, multi-line `<table>` HTML, etc., detected as any rendered child whose stripped output contains a `\n`), the block is hoisted out of the list item and emitted at column 0 on the line immediately following the list item's inline content. Adjacent hoisted blocks are separated by a blank line so Obsidian renders them as separate blocks. The list is implicitly terminated at column 0 тАФ the hoisted block renders as a top-level element, not as visual content of the list item. Each occurrence is logged to `migration-report.md` under `## Warnings` with the page title тАФ the original Confluence relationship ("this block belongs to this list item") is lost in conversion and the page may need manual review. The list marker line **always keeps its trailing space** after the marker (`- ` for bullets, `1. ` for ordered) even when the list item's inline content is empty тАФ CommonMark requires a space after the marker for it to be recognized as a list bullet (a bare `-` on its own line parses as plain text, not a list item), so the empty-body-followed-by-hoisted-block case (e.g. an `<li>` containing only an `expand` macro) emits `- ` on the marker line and the hoisted callout on the next line, preserving the visual "empty bullet + block" intent.
- **Inline HTML preserved**: `<sub>` and `<sup>` are emitted as raw HTML (`<sub>2</sub>`, `<sup>2</sup>`) тАФ Markdown has no native equivalent and Obsidian renders them in reading view. Same approach as the existing `<u>` handling.
- **Plain-text Markdown escape**: in user-visible body text, the characters `\`, `` ` ``, `*`, `_`, `~`, `#`, `[`, `]`, and `$` are each prefixed with a backslash (e.g. `\` тЖТ `\\`, `` ` `` тЖТ `` \` ``, `*` тЖТ `\*`, `_` тЖТ `\_`, `~` тЖТ `\~`, `#` тЖТ `\#`, `[` тЖТ `\[`, `]` тЖТ `\]`, `$` тЖТ `\$`) so Obsidian renders them as literal characters instead of triggering Markdown syntax (inline code / bold / italic / strikethrough / tags / links / footnotes / task lists / math). The escape over-covers a few low-risk positional cases тАФ intraword `_` and mid-line `#` are technically safe in CommonMark, but are escaped anyway for simplicity and to defeat Obsidian's tag extension (`#123` would otherwise become a tag). Source `.md` files will show backslash-prefixed characters more often as a result; rendered output is unchanged. Applied to text nodes whose ancestor chain includes any of `<p>`, `<h1>`тАУ`<h6>`, `<li>`, `<strong>`, `<em>`, `<u>`, `<sub>`, `<sup>`, or an uncolored `<span>` (i.e. a `<span>` whose `style` does not contain `color:` тАФ colored spans become raw `<font>` HTML, where Obsidian does not process Markdown escapes, so their inner text is left verbatim). The escape is **suppressed** whenever a closer ancestor is `<code>`, `<pre>`, `ac:plain-text-body`, or `ac:plain-text-link-body` (verbatim contexts тАФ inline code, code-fence and latex CDATA bodies). Inside `<a>` and `<ac:link>` (link display text), only `[` and `]` are escaped тАФ the other characters stay verbatim so inline formatting and math survive inside the link (e.g. `[**bold link**](url)` and `[$x^2$](url)` still render correctly), while the brackets escape so a user-typed `[draft]` inside link text doesn't break the constructed `[text](url)` parse into `[[draft]](url)`. Table-cell text follows the separate Tables rule below (no plain-text escape applied). Already-escaped sequences are not detected: a Confluence-typed literal `\]` becomes `\\\]` (four characters) in the output, which Obsidian renders back to `\]` тАФ preserving the original two-character text.
- **Inline `<code>`**: always emitted as raw HTML `<code>X</code>` (not Markdown backticks). Keeps rendering consistent with colored inline code below.
- **Block `<pre>`**: not a code block тАФ emitted as a sequence of inline `<code>` elements, one per rendered line. Confluence syntax-highlighted snippets carry colors via nested `<span style="color:...">` children which die inside a Markdown fenced code block (Markdown doesn't process HTML inside `` ``` ``); multi-`<code>` lets the existing colored-span тЖТ `<font>` rule compose so highlight colors survive. Lines come from splitting the rendered child content on `\n` **and** `<br/>` (both produce line breaks identically). Leading and trailing empty lines are trimmed; internal empty lines are preserved as `<code></code>` placeholders (vertical spacing inside the snippet matters тАФ e.g. paragraph breaks in HTTP transcripts, log dumps). An empty `<pre>` (no content, or only whitespace) emits nothing. Each line's children render through the **full document-mode Markdown converter** тАФ colored-span тЖТ `<font>` and uncolored-span unwrap compose naturally; `<u>` / `<sub>` / `<sup>` survive as raw HTML (Obsidian renders HTML inside inline `<code>`); but Markdown-output tags (`<strong>` тЖТ `**foo**`, `<em>` тЖТ `*foo*`, `<a>` тЖТ `[text](u)`) render literally inside the `<code>` because Obsidian does NOT process Markdown inside inline `<code>` тАФ accepted as a known leak; non-span inline HTML inside `<pre>` is rare in Confluence content. Visual line separation relies on Obsidian's "Strict line breaks" setting being OFF (the default), under which the soft break between two consecutive inline `<code>` elements becomes a rendered `<br>`. With strict-line-breaks ON, all lines collapse onto a single visual line тАФ not supported. The rule applies at document level only. A `<pre>` encountered inside a `<td>` / `<th>` falls through to the cell's existing raw-HTML pass-through (the `<pre>` survives as a literal `<pre>` element in the cell) and is logged to `migration-report.md` with the page title for manual review тАФ table-cell rendering is its own world and is not extended for this corner case. (Was previously a fenced code block тАФ switched to multi-`<code>` to preserve syntax-highlighting colors that fenced blocks strip.)
- **Text color**: a `<span>` whose `style` attribute contains `color:` becomes `<font style="<original style>">X</font>` тАФ the entire `style` attribute is copied verbatim. Matches the Obsidian font-color plugin convention. Spans without a `color:` declaration keep the existing strip-wrapper behavior (children rendered, no wrapper). Composes with the inline-`<code>` rule above when the source is `<code><span style="color: rgb(...);">X</span></code>` тЖТ `<code><font style="color: rgb(...);">X</font></code>` (the natural nesting carries through). The **reverse source nesting** тАФ `<span style="color: rgb(...);"><code>X</code></span>` тАФ is normalized to the same code-outer shape: when a colored span's children consist of **exactly one `<code>` element** (ignoring purely-whitespace text nodes), the span/code pair is rewritten as `<code><font style="color: rgb(...);">X</font></code>` тАФ `<code>` ends up outermost regardless of which order Confluence's editor emitted. Obsidian's font-color plugin only colors text reliably when the `<font>` sits *inside* the monospace `<code>` block; the other nesting (`<font><code>X</code></font>`) leaves the code text in its default colour because `<code>`'s monospace styling supersedes the surrounding `<font>` color. The normalization is **narrow**: it only fires for sole-`<code>`-child spans. A colored span with mixed content (`<span style="">prefix <code>X</code> suffix</span>`) or multiple `<code>` children stays as `<font>...</font>` wrapping the whole thing тАФ restructuring those would re-parent siblings and can break adjacent text layout, and the case is unusual in Confluence content.
- **Macro: Children Display** тЖТ inserted only when the Confluence page explicitly contains the macro; renders as a Dataview LIST block. Two forms:
  - **No `page` parameter**: scoped to this page's children тАФ `WHERE file.folder = this.file.folder + "/" + this.file.name`
  - **With `page` parameter** (e.g., `<ri:page ri:content-title="Computer"/>`): scoped to the referenced page's children тАФ `WHERE file.folder = [[Computer]].file.folder + "/" + [[Computer]].file.name`. The wiki-link uses the sanitized/collision-resolved title (same resolver as internal page links). If the referenced page is not in the migrated title map, the macro is dropped and a broken-reference entry is added to `migration-report.md`.
- **Heading levels**: `<h1>` тЖТ `#`, `<h2>` тЖТ `##`, `<h3>` тЖТ `###`; `<h4>`, `<h5>`, `<h6>` all clamp to `######` (the deepest Markdown heading)
- **Macro: Excerpt** тЖТ rendered Markdown body wrapped in a fenced `` ```excerpt `` code block followed immediately by a `^excerpt` block anchor on the next line; rendered by a custom Obsidian plugin
- **Macro: Include Excerpt** тЖТ `![[Page Title#^excerpt]]` Obsidian block transclusion; page title extracted from `ac:parameter[ac:name=""]` тЖТ `ri:page[@ri:content-title]`; `nopanel` parameter ignored
- **Macro: `latex-inline`** / **`latex-block`** / **`latex`** тЖТ `$<CDATA>$` (Obsidian inline math). All three variants produce the same `$X$` math wrapping тАФ `latex-block` is treated as an alias for `latex-inline`, not as Obsidian block math (`$$X$$`). The only difference is positional: `latex-block` emits a trailing `\n` after the closing `$` (output `$X$\n`) so consecutive `latex-block` macros land on separate source lines instead of gluing together; `latex-inline` emits no trailing newline (output `$X$`) so it flows within prose. Body normalization runs in two steps: (1) any run of newlines in the CDATA body (together with whitespace on either side) is collapsed to a **single ASCII space** тАФ adjacent lines are joined with one space between them (Obsidian's inline math renderer breaks on embedded newlines; LaTeX is whitespace-insensitive between tokens, so `\sum\nx` тЖТ `\sum x` renders identically to the source and a multi-line `\text{some\ntext}` correctly becomes `\text{some text}`); (2) every remaining whitespace character (`\s` тАФ Unicode whitespace such as NBSP `U+00A0`, em-space `U+2003`, narrow no-break space `U+202F`, etc., plus stray ASCII tabs / vertical tabs / form feeds) is replaced **one-to-one** with an ASCII space `U+0020`. Confluence's editor frequently inserts NBSPs where the author typed an ordinary space (Shift+Space, paste-from-formatted-source), and these visibly differ тАФ or fail to parse тАФ in Obsidian's math renderer. Same broad `\s`-to-space normalization is used for `ac:image` filenames (see Tables). One edge case where space-separator is technically wrong: a Confluence author who line-breaks immediately before a subscript / superscript marker (`\sum\n_{i=1}^n`) gets a separating space (`\sum _{i=1}^n`), which LaTeX may parse as `\sum` followed by an independent `_{...}` rather than as `\sum`'s subscript тАФ accepted because the source corpus contains no multi-line equations of this shape. (Was previously split: `latex-block` rendered as `$$X$$` with the multiline body preserved verbatim. Collapsed into `latex-inline` because every `latex-block` instance in the source corpus contains a single-line equation тАФ the block-vs-inline distinction added no value and forfeited inline positioning within prose. Re-introduce the split if the corpus ever grows multi-line equations: matrices via `\begin{matrix}тАж\end{matrix}`, aligned environments via `\begin{aligned}тАж\end{aligned}`, etc.)
- **Macro: `code`** тЖТ fenced code block with language from `ac:parameter[language]`
- **Macro: `info`** тЖТ `> [!info]` callout wrapping the body content
- **Macro: `warning`** тЖТ `> [!warning]` callout wrapping the body content
- **Macro: `expand`** / **`ui-expand`** тЖТ native Obsidian callout, foldable and collapsed by default: `> [!expand]- TITLE` (input macro `expand`) or `> [!expand-ui]- TITLE` (input macro `ui-expand`), with the body on subsequent `> `-prefixed lines. Body is rendered through the **full document-mode Markdown converter** тАФ lists, code blocks, headings, callouts, internal links, nested tables, nested expands all render natively (Obsidian re-parses callout contents as Markdown) тАФ then stripped, 3+ consecutive newlines collapsed to 2, every line prefixed with `> ` (empty lines become bare `>`). Nesting falls out for free тАФ an inner expand's own `> ` prefixes become `> >` under the outer one. When `title` is absent, the callout header defaults to `Click here to expand...` for both macros. Empty body still emits the header line (never dropped). Note the asymmetric output naming: input macro `ui-expand` тЖТ output callout type `expand-ui`. Visual differentiation between `expand` and `expand-ui` depends on **custom CSS** on `[data-callout="expand"]` and `[data-callout="expand-ui"]` in the destination vault тАФ Obsidian has no built-in styling for these callout type names; without the CSS, both render as plain "note" callouts. Cell-context expand keeps the inline `<details><summary>тАж</summary>тАж</details>` form (cells can't host block-level callouts) тАФ see the **Tables** decision. (Was previously fenced blocks for a custom Obsidian plugin тАФ switched to native callouts to drop the plugin dependency; callouts re-parse Markdown internally the same way the plugin did.)
- **Macro: `anchor`** тЖТ silently dropped (Obsidian has no equivalent free-floating anchor; not logged as unknown)
- **Macro: `toc`** тЖТ silently dropped (Obsidian's built-in Outline pane renders the heading tree natively)
- **Macro: `view-file`** / **`viewpdf`** / **`multimedia`** тЖТ `![[PageName/filename]]` where filename comes from the embedded `ri:attachment`; the attachment is downloaded into the per-page subfolder (same flow as inline images); Obsidian renders PDFs, video, and audio inline natively in reading view
- **Macro: `ui-tabs`** тЖТ fenced block for the **["Tabs" Obsidian plugin](https://github.com/ycnmhd/obsidian-tabs)**, with the literal opening fence `~~~tabs`, one `---tab <TITLE>` separator per child `ui-tab`, and the literal closing fence `~~~`. Preserves horizontal-tab semantics in Obsidian's reading view. Assembled by the `ui-tabs` renderer itself (not by delegating to per-tab rendering): it scans direct `ui-tab` descendants of its `ac:rich-text-body` in document order, rendering each tab's `title` parameter on a `---tab TITLE` line and the tab's `ac:rich-text-body` immediately below (no blank line between). Tab body is rendered through the **full document-mode Markdown converter** тАФ lists, code blocks, callouts, internal links, nested tables, nested expands all work natively inside a tab; each body is stripped of leading/trailing whitespace. Non-`ui-tab` content inside the `ui-tabs` body is dropped silently (matches Confluence's visual behaviour тАФ its tab UI ignores stray content too; cursor-parking `<p><br/></p>` paragraphs are the common case). A `ui-tabs` with zero `ui-tab` children produces no output (the whole block is dropped). `ui-tab` is **not** dispatched by `_render_macro` тАФ it appears only as a child of `ui-tabs` and is rendered inline by the parent; an orphan `ui-tab` outside a `ui-tabs` is an invariant violation and would surface as an unknown-macro entry in `migration-report.md`. (Was previously stacked `<details>` collapsibles тАФ switched to the plugin format to recover horizontal tab layout.)
- **Macro: `widget`** тЖТ `![](url)` Media Extended embed; URL extracted from `ac:parameter[ac:name="url"]` тЖТ `ri:url[@ri:value]`; used in practice only for YouTube embeds
- **Macro: `section`** тЖТ entire macro and body dropped (no output, not logged as unknown); used for multi-column layouts that have no Markdown equivalent
- **Macro: `details`** тЖТ entire macro and body dropped (no output, not logged as unknown); Confluence Page Properties metadata has no Markdown equivalent
- **Macro: `pagetree`** / **`pagetreesearch`** / **`livesearch`** тЖТ silently dropped; Obsidian's file explorer and global search cover these natively
- **Macro: `recently-updated`** тЖТ Dataview query `LIST FROM "" SORT modified DESC LIMIT <max>` where `<max>` comes from the `max` parameter (default `15`); sorts by the `modified` frontmatter field (Confluence-recorded timestamp); `spaces`, `types`, and `theme` parameters are ignored
- **Internal page link** тЖТ `[[Page Title|Display Text]]` Obsidian wiki link (page titles are unique across the instance)
- **External URL link** тЖТ `[Display Text](url)` standard Markdown link
- **YouTube embed** (`<ri:url>`) тЖТ `![](url)` Media Extended plugin syntax
- **Attachments**: downloaded into per-page subfolder `<Space>/<Parent>/<PageName>/<filename>`, referenced as `![[PageName/filename]]`; inline images embedded in page body (`<ac:image><ri:attachment>`) are treated identically тАФ downloaded to the same subfolder and emitted as `![[PageName/filename]]` at the point of the image. Inline-image sizes carry over via the **Obsidian wiki-link size suffix** only when the source `<ac:image>` carries an `ac:width` attribute: `ac:width="W"` alone тЖТ `![[PageName/filename|W]]` (Obsidian scales height proportionally); `ac:width="W"` plus `ac:height="H"` тЖТ `![[PageName/filename|WxH]]`; neither attribute (or `ac:height` alone) тЖТ bare `![[PageName/filename]]`. `ac:height` is intentionally **ignored when `ac:width` is absent** тАФ Obsidian's `|0xH` "height-only" convention does not render reliably in practice (image either renders unsized or with the wrong aspect ratio), so the safer fallback is to drop the size hint and let Obsidian display the image at its natural size. Values are read verbatim from the attributes тАФ no parsing, casting, or unit-stripping (Confluence's documented format is integer pixels; any non-integer value passes through and may render wrong, but that mirrors the cell `<img>` rule below). The `ac:style` attribute (e.g. `max-height: 250.0px;`) is **never** parsed тАФ Confluence's editor sets `ac:width` / `ac:height` alongside any inline style, so attribute-only reading is sufficient in practice. External-URL images (`<ac:image><ri:url>`) stay as `![](url)` with no size тАФ Markdown image syntax has no native size suffix. Any Unicode whitespace in the filename is normalized **one-to-one** to ASCII space `U+0020` (same `\s`-to-space pattern used for `ac:image` in table cells and for latex bodies) at **both** sites that touch the filename: every wiki-link reference emitted by the converter, **and** the on-disk filename when the attachment is written. Both must agree тАФ Confluence's editor sometimes embeds NBSP (`U+00A0`) etc. in attachment filenames (most common: screenshots whose default names contain NBSP-separated timestamps), and without paired normalization Obsidian can't resolve the link to the file on disk. Applies to all reference paths: inline `<ac:image>`, attachment `<ac:link>`, and the `view-file` / `viewpdf` / `multimedia` macros.
- **Unknown macros**: logged and skipped; a `migration-report.md` written to vault root with macro name, page, and count
- **Tables**: every `<table>` is emitted as cleaned, prettified raw HTML (no Markdown pipe tables тАФ Obsidian doesn't render Markdown inside `<table>`). Cleaning: strip `<colgroup>` and its contents; on every element, drop all attributes except `colspan`, `rowspan`, `data-highlight-colour`, and `style` (kept verbatim); on `<td>`/`<th>`, `data-highlight-colour="X"` is appended to the cell's `style` as `background-color: X;` (merging with any existing `style`); the colour name `grey` is remapped to `#F4F5F7`. Every `<th>` with no explicit background-color (no `data-highlight-colour`, no `background-color:` declaration in `style`) receives an automatic `background-color: #F4F5F7;` (Confluence's default header tint). Cell contents stay as verbatim Confluence HTML with these exceptions: `<ac:link>` is replaced by plain text (the `<ac:plain-text-link-body>` CDATA if present, else the `ri:page ri:content-title` value); `<ac:structured-macro ac:name="expand">` / `ac:name="ui-expand"` is converted to an inline single-line `<details><summary>TITLE</summary>CONTENT</details>` (no blank lines, no newlines between elements) тАФ `TITLE` from `ac:parameter[ac:name="title"]` falling back to `Click here to expand...` for `expand` when absent; `CONTENT` is the `ac:rich-text-body` rendered recursively under these same table-cell rules (so nested `ac:*` keeps being escaped, nested `ac:link`/`ac:image`/`expand`/`ui-expand` apply their cell exceptions, nested expands stay inline single-line); every other `<ac:*>` element is serialized back to its source XML and HTML-escaped (`<` тЖТ `&lt;`, `>` тЖТ `&gt;`) so it appears as visible literal text in the rendered cell тАФ a manual-cleanup signal тАФ with one further exception: `<ac:image>` containing a `<ri:attachment ri:filename="X">` is converted to `<img src="X" />` (bare filename тАФ no per-page subfolder prefix; may need manual fixup; any Unicode whitespace in the filename is normalized to ASCII space), and `ac:width` / `ac:height` (if present) carry over as `width` / `height` attributes; all other `ac:image` attributes are dropped. An `<ac:image>` lacking a usable `ri:filename` falls back to the escape-as-source-XML treatment and is logged to `migration-report.md` with the page title. Three wrapper-stripping rules apply inside cells: `<div class="content-wrapper">` is always unwrapped (Confluence noise); a `<p>` whose only content is one or more such escaped `ac:*` macros is unwrapped; and a `<p>` whose only non-whitespace content is one or more `<br/>` tags is **dropped entirely** (Confluence emits these as cursor-parking spots тАФ e.g. `<p class="auto-cursor-target"><br /></p>` тАФ and they would otherwise leave visible vertical noise in the cell). Attributes on the `<p>` are ignored when matching this rule. Prettification: 4-space indent, one element per line тАФ except inside `<td>`/`<th>`, where children render inline; raw text nodes inside a `<td>` that already contain literal newlines preserve them. When a `<table>` appears inside another table's cell, the inner table is recursively rendered with the same prettification rules, indented at parent-cell-depth + 1, **but with no `\n` before the opening `<table>` or after the closing `</table>`** тАФ the nested table sits flush against any preceding/following content in the parent cell (`BEFORE<table>...</table>AFTER`). The inner table keeps its internal multi-line structure for rows and cells; only the boundary newlines against the parent cell are removed. Asymmetric: the opening `<table>` has no leading indent (it's inline), but the closing `</table>` keeps its depth-aligned indent тАФ visible only in source HTML, irrelevant to Obsidian's renderer. Followed by a blank line per the block-spacing rule (except when nested inside another cell тАФ those stay flush as described above).
- **Filename sanitization**: characters illegal in filenames (`:`, `/`, `?`, `*`, `<`, `>`, `|`, `\`) in page titles are replaced with their visually similar Unicode fullwidth equivalents (`я╝Ъя╝Пя╝Яя╝Кя╝Ья╝Юя╜Ья╝╝`) when used as file or directory names тАФ preserves the appearance and is lossless (no two distinct Confluence titles collapse to the same on-disk name); post-sanitization collisions are still resolved by suffixing the Confluence page ID and logged in `migration-report.md`
- **Migration mode**: one-time, full re-run on every invocation (no incremental sync, no resume); Ctrl+C triggers a graceful stop that writes `migration-report.md` before exiting
- **migration-report.md**: written to vault root on graceful stop, crash, or completion; contains all unknown macros encountered (macro name, page title, count) and all per-page failures (page title, error reason)
- **Frontmatter**: title, created, modified, author, confluence_url, labels (as Obsidian tags)
- **Config**: `.env` file (`CONFLUENCE_URL`, `CONFLUENCE_USER`, `CONFLUENCE_PASS`, `VAULT_PATH`, `CONFLUENCE_VERIFY_SSL`, `DOWNLOAD_ATTACHMENTS`); space keys passed as CLI args
- **Progress display**: each page migration prints `[<n>/<total>] <Page Title>` to stdout
- **SSL**: verification skipped (`verify=False`), warnings suppressed; controlled by `CONFLUENCE_VERIFY_SSL` env var
- **Attachment download**: controlled by `DOWNLOAD_ATTACHMENTS` env var (default `true`); when `false`, attachments are skipped and attachment references are emitted as broken wiki-links (e.g., `![[PageName/diagram.png]]`)
